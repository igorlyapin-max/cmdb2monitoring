Техническое задание
Проект: cmdb2monitoring
Версия: 0.1
Дата: 2026-04-29

1. Назначение проекта

Проект cmdb2monitoring предназначен для интеграции CMDBuild с внешними системами мониторинга и обработки событий.
Разработка ведется в одном репозитории с разделением по микросервисам.

Первый микросервис: cmdbwebhooks2kafka.
Второй микросервис: cmdbkafka2zabbix.
Третий микросервис: zabbixrequests2api.
Frontend/BFF: monitoring-ui-api.

Назначение cmdbwebhooks2kafka:
- принимать webhook-события из CMDBuild;
- нормализовать входные данные;
- различать события create, update, delete;
- публиковать события в Kafka;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение cmdbkafka2zabbix:
- читать нормализованные события CMDBuild из Kafka;
- читать правила конвертации из Git-managed JSON-файла;
- применять регулярные выражения для валидации и выбора групп, шаблонов и интерфейсов Zabbix;
- формировать JSON-RPC payload, принимаемый Zabbix API;
- публиковать сформированный Zabbix JSON-RPC payload в отдельный Kafka topic;
- сохранять идентификатор последнего обработанного объекта в локальный state-файл;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение zabbixrequests2api:
- читать JSON-RPC запросы к Zabbix из Kafka;
- валидировать обязательные поля Zabbix payload;
- проверять наличие host groups, templates и template groups в Zabbix;
- выполнять вызовы Zabbix API;
- публиковать результат или ошибку в отдельный Kafka response topic;
- обрабатывать объекты "gentle" режимом с настраиваемой задержкой между объектами;
- сохранять идентификатор последнего обработанного объекта в локальный state-файл;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение monitoring-ui-api:
- предоставлять Node.js frontend и backend-for-frontend для оператора интеграции;
- отображать health/status микросервисов;
- принимать login/password для CMDBuild и Zabbix, если IdP отключен;
- хранить credentials только в server-side session;
- предоставлять console настройки IdP/SAML2 после авторизации;
- загружать и валидировать JSON rules-файл;
- выполнять dry-run правил конвертации;
- синхронизировать catalog из Zabbix: templates, host groups, template groups, known tags;
- синхронизировать catalog из CMDBuild: классы, атрибуты, lookup-значения;
- хранить catalog cache локально без секретов.

2. Технологический стек

- Язык: C#.
- Платформа: .NET.
- Тип сервиса: ASP.NET Core Web API / Minimal API.
- Брокер сообщений: Apache Kafka.
- CMDB: CMDBuild.
- Логи: structured JSON logs, временно через Kafka-топик, в будущем через ELK.
- Репозиторий: monorepo.
- Frontend/BFF: Node.js без прямого доступа браузера к Kafka/CMDBuild/Zabbix.

4. Микросервис cmdbwebhooks2kafka

4.1. HTTP endpoint

Микросервис должен принимать POST-запросы от CMDBuild:

POST /webhooks/cmdbuild

Health endpoint:

GET /health

4.2. Входной формат webhook

CMDBuild должен отправлять JSON body с плоской структурой:

{
  "source": "cmdbuild",
  "eventType": "create|update|delete",
  "cmdbuildEvent": "card_create_after|card_update_after|card_delete_after",
  "className": "Notebook|PC|Server|tk",
  "id": "{server:Id}",
  "code": "{server:Code}",
  "ip_address": "{server:IPAddress}",
  "description": "{server:Description}",
  "os": "{server:OS}",
  "zabbixTag": "{server:zabbixTag}"
}

Поля могут быть расширены в будущем без изменения базового контракта.

По результатам проверки CMDBuild корректно подставляет:
- {server:Id} в поле id;
- {server:Code} в поле code.
- {server:OS} в поле os;
- {server:zabbixTag} в поле zabbixTag.

Особенность CMDBuild lookup-полей:
- для lookup OS webhook передает внутренний numeric id, а не текстовое значение;
- для lookup zabbixTag webhook передает внутренний numeric id, а не текстовое значение;
- rules-файл второго микросервиса должен уметь маппить эти id в смысловые значения для Zabbix.

4.3. Поддерживаемые события CMDBuild

CMDBuild webhook events:
- card_create_after -> eventType=create;
- card_update_after -> eventType=update;
- card_delete_after -> eventType=delete.

4.4. Поддерживаемые классы CMDBuild

Базовый класс в CMDBuild: Computer.

Так как Computer является prototype-классом, webhook настраивается на конкретные наследники:
- Notebook;
- PC;
- Server;
- tk.

Для каждого класса должны быть заведены webhook-записи на create, update, delete.

Для подклассов Computer используются дополнительные атрибуты:
- OS: lookup со значениями Windows, Linux, Windows server, vmware;
- zabbixTag: lookup со значениями tag1, tag2, tag3.

4.5. Заголовки webhook из CMDBuild

CMDBuild webhook должен передавать только:

Authorization: Bearer <token>

Content-Type вручную в CMDBuild webhook headers не указывать.
Причина: CMDBuild сам добавляет Content-Type. Если прописать его вручную, запрос приходит как application/json,application/json.

Микросервис должен быть устойчив к некорректному/дублированному Content-Type и читать JSON body вручную.

5. Авторизация webhook

Требуется Bearer token авторизация входящих webhook-запросов.

Режимы токена:
- static: токен хранится в конфиге микросервиса;
- dynamic: заглушка под будущую реализацию, источник динамического токена пока не определен.

На стороне CMDBuild токен прописывается вручную в webhook header Authorization.

Важно: секретное значение токена не должно храниться в git в production-конфиге. Для реального окружения использовать переменные окружения, секрет-хранилище или local config, исключенный из git.

6. Kafka

6.1. Топик бизнес-событий

Микросервис публикует нормализованные CMDBuild webhook-события в Kafka.

Development topic:

cmdbuild.webhooks.dev

Production/base config topic:

cmdbuild.webhooks

6.2. Текущий режим подключения Kafka

Текущая локальная Kafka работает без авторизации:
- SecurityProtocol: Plaintext;
- SASL отключен;
- TLS отключен;
- username/password не используются.

Адреса Kafka:
- с хоста: localhost:9092 или 127.0.0.1:9092;
- из Docker-сети: kafka:29092.

6.3. Kafka security config

Все параметры подключения должны задаваться через конфиг:
- BootstrapServers;
- Topic;
- ClientId;
- SecurityProtocol;
- SaslMechanism;
- Username;
- Password;
- Acks;
- EnableIdempotence;
- MessageTimeoutMs.

Сейчас используется Plaintext. В будущем должна быть возможность включить SASL/TLS через конфиг без изменения кода.

6.4. Kafka message contract

Микросервис публикует envelope:

{
  "source": "cmdbuild",
  "eventType": "create|update|delete",
  "entityType": "Notebook|PC|Server|tk",
  "entityId": "<id или code>",
  "receivedAt": "<UTC timestamp>",
  "payload": { ... исходный webhook payload ... }
}

Правило entityId:
- сначала читать поле id;
- затем code;
- затем entityId.

Kafka key:
- entityId, если найден;
- иначе eventType.

6.5. Kafka topics

Микросервис не создает Kafka topics при старте.
Необходимые topics должны быть созданы заранее средствами инфраструктуры.

Обязательные topics:
- cmdbuild.webhooks.dev для dev-окружения;
- cmdbuild.webhooks для production/base config;
- cmdbwebhooks2kafka.logs.dev для dev-логов;
- cmdbwebhooks2kafka.logs для production/base config логов.

7. Логирование и ELK

ELK-сервера пока нет.

Требуется заложить настройки и код так, чтобы позже подключить ELK без изменения бизнес-логики.

Текущий временный режим:
- структурированные логи пишутся в Kafka;
- формат логов: JSON;
- отдельный Kafka-топик для логов.

Development log topic:

cmdbwebhooks2kafka.logs.dev

Production/base config log topic:

cmdbwebhooks2kafka.logs

Конфиг должен содержать секцию ElkLogging:
- Enabled;
- Mode;
- Elk.Endpoint;
- Elk.Index;
- Elk.ApiKey;
- Kafka настройки для лог-топика;
- MinimumLevel;
- ServiceName;
- Environment;
- FlushTimeoutMs.

8. Конфигурация

Все настройки микросервиса должны храниться в конфиге:
- Service;
- CmdbWebhook;
- Kafka;
- ElkLogging;
- Logging.

Не допускается хардкодить:
- Kafka bootstrap servers;
- Kafka topics;
- auth/security settings;
- webhook route;
- имена полей для определения eventType/entityType/entityId;
- ELK/Kafka logging settings.

9. Текущие локальные настройки CMDBuild

CMDBuild доступен локально:

http://localhost:8090/cmdbuild

Учетная запись для настройки в dev:

admin/admin

REST base URL:

http://localhost:8090/cmdbuild/services/rest/v3

Webhook URL для текущего локального сервиса:

http://192.168.202.100:5080/webhooks/cmdbuild

Если сервис запускается в другом окружении, URL необходимо заменить в CMDBuild webhook settings.

В текущем dev-окружении заведены webhook-записи:
- для классов Notebook, PC, Server, tk;
- для событий card_create_after, card_update_after, card_delete_after;
- body содержит поля id, code, ip_address, description, os, zabbixTag.

10. Проверки приемки

10.1. HTTP

- GET /health возвращает статус ok.
- POST /webhooks/cmdbuild принимает корректный JSON.
- Некорректный JSON возвращает 400.
- Дублированный Content-Type не должен ломать обработку, если тело запроса является JSON.

10.2. CMDBuild -> Kafka

Для Notebook, PC, Server, tk:
- create в CMDBuild приводит к сообщению в Kafka;
- update в CMDBuild приводит к сообщению в Kafka;
- delete в CMDBuild приводит к сообщению в Kafka.

В Kafka payload должен содержать:
- id;
- code;
- ip_address;
- description;
- os;
- zabbixTag;
- className;
- eventType;
- cmdbuildEvent.

10.3. Kafka

- Сообщения публикуются в cmdbuild.webhooks.dev в dev-окружении.
- Логи публикуются в cmdbwebhooks2kafka.logs.dev в dev-окружении.
- Необходимые Kafka topics созданы заранее до запуска микросервиса.

10.4. Логи

Лог-события должны фиксировать:
- получение webhook;
- eventType;
- entityType;
- entityId;
- успешную публикацию в Kafka с topic/partition/offset;
- ошибки разбора JSON;
- ошибки Kafka publishing.

13. Микросервис cmdbkafka2zabbix

13.1. Назначение

Микросервис cmdbkafka2zabbix является вторым этапом пайплайна:

CMDBuild -> cmdbwebhooks2kafka -> Kafka topic cmdbuild.webhooks.* -> cmdbkafka2zabbix -> Kafka topic zabbix.host.requests.*.

Микросервис не вызывает Zabbix API напрямую на текущем этапе.
Он формирует JSON-RPC request, совместимый с Zabbix host.create, и публикует его в Kafka.

13.2. Kafka input

Development input topic:

cmdbuild.webhooks.dev

Production/base input topic:

cmdbuild.webhooks

Consumer group должен задаваться через конфиг.
Kafka offsets должны коммититься только после успешной обработки сообщения или после осознанного skip некорректного/неподдерживаемого события.

13.3. Kafka output

Development output topic:

zabbix.host.requests.dev

Production/base output topic:

zabbix.host.requests

В output topic публикуется тело сообщения в виде JSON-RPC payload для Zabbix, например method=host.create.
Kafka message key:
- entityId, если он есть;
- иначе имя Zabbix host;
- иначе eventType.

Микросервис не создает Kafka topics при старте.
Input/output/log topics должны быть созданы заранее средствами инфраструктуры.

13.4. Правила конвертации из Git

Правила конвертации хранятся в JSON-файле:

rules/cmdbuild-to-zabbix-host-create.json

Источник правил настраивается через конфиг:
- RepositoryPath;
- RulesFilePath;
- PullOnStartup;
- GitExecutablePath;
- TemplateName.

В dev-режиме RepositoryPath указывает на текущий monorepo.
В будущем RepositoryPath может быть заменен на отдельную Git working copy с правилами.

13.5. Данные для создания host в Zabbix

Для Zabbix host.create должны быть сформированы:
- host: техническое имя хоста;
- name: отображаемое имя;
- interfaces: IP, тип интерфейса, порт;
- groups: массив groupid;
- templates: массив templateid;
- tags: массив tag/value;
- status;
- inventory_mode.

Zabbix API принимает groupid/templateid, поэтому человекочитаемые имена групп и шаблонов в rules-файле должны храниться вместе с реальными ID из Zabbix.

Локально подтвержден Zabbix:
- URL UI: http://localhost:8081;
- API endpoint: http://localhost:8081/api_jsonrpc.php;
- версия: 7.0.25;
- dev user/password: Admin/zabbix.

Используемые локальные справочники:
- group Linux servers: groupid=2;
- group Discovered hosts: groupid=5;
- group Virtual machines: groupid=6;
- group Hypervisors: groupid=7;
- group Applications: groupid=19;
- group Databases: groupid=20;
- template Linux by Zabbix agent: templateid=10001;
- template Windows by Zabbix agent: templateid=10081;
- template ICMP Ping: templateid=10564;
- template Generic by SNMP: templateid=10563;
- template Network Generic Device by SNMP: templateid=10226.

Используемые CMDBuild lookup id для dev-окружения:
- OS Windows: 105130;
- OS Linux: 105146;
- OS Windows server: 105152;
- OS vmware: 105161;
- zabbixTag tag1: 106857;
- zabbixTag tag2: 106852;
- zabbixTag tag3: 106850.

Rules-файл должен поддерживать как человекочитаемые значения, так и numeric lookup id, потому что webhook CMDBuild передает lookup-поля в виде id.
При создании host в Zabbix поля OS и zabbixTag должны преобразовываться в Zabbix tags:
- cmdb.os=<Windows|Linux|Windows server|vmware>;
- cmdb.zabbixTag=<tag1|tag2|tag3>.

13.6. Регулярные выражения и выбор параметров

Regex используется не только для валидации, но и для выбора параметров Zabbix.

В rules-файле должны быть отдельные блоки:
- validationRegex для проверки входных данных, например ip_address;
- groupSelectionRules для выбора групп Zabbix;
- templateSelectionRules для выбора шаблонов мониторинга;
- interfaceSelectionRules для выбора типа интерфейса;
- tagSelectionRules для выбора Zabbix tags;
- normalization.hostName.regexReplacements для нормализации имени host.

Правила применяются по priority.
Для групп и шаблонов допускается объединение нескольких совпавших правил с дедупликацией по groupid/templateid.
Для интерфейса выбирается первое совпавшее правило, иначе fallback.

Текущие правила выбора по OS:
- Linux или lookup id 105146 -> группа Linux servers, шаблон Linux by Zabbix agent, tag cmdb.os=Linux;
- Windows или lookup id 105130 -> группа Discovered hosts, шаблон Windows by Zabbix agent, tag cmdb.os=Windows;
- Windows server или lookup id 105152 -> группа Discovered hosts, шаблон Windows by Zabbix agent, tag cmdb.os=Windows server;
- vmware или lookup id 105161 -> группа Hypervisors, шаблон Zabbix agent, tag cmdb.os=vmware.

Текущие правила выбора по zabbixTag:
- tag1 или lookup id 106857 -> tag cmdb.zabbixTag=tag1;
- tag2 или lookup id 106852 -> tag cmdb.zabbixTag=tag2;
- tag3 или lookup id 106850 -> tag cmdb.zabbixTag=tag3.

13.7. T4-шаблон

Формирование итогового JSON-RPC payload должно выполняться полноценным T4 engine из rules-файла.

Используемый runtime engine:

Mono.TextTemplating

Текущий шаблон:

t4Templates.hostCreateJsonRpcRequestLines

Настройки T4 engine задаются в конфиге микросервиса:
- ConversionRules.TemplateEngine;
- ConversionRules.TemplateName;
- ConversionRules.AddDefaultDirectives.

Если шаблон в rules-файле не содержит собственную директиву <#@ template ... #>, сервис добавляет стандартные directives автоматически:
- template language=C#;
- assembly reference на сборку микросервиса;
- import namespace CmdbKafka2Zabbix.Conversion;
- parameter Model типа CmdbKafka2Zabbix.Conversion.ZabbixHostCreateModel.

Если в будущем rules-файл будет содержать полностью самостоятельный T4-шаблон со своими directives, автоматическое добавление directives можно отключить через ConversionRules.AddDefaultDirectives=false.

Шаблон должен формировать JSON-RPC request с:
- jsonrpc=2.0;
- method=host.create;
- params.host;
- params.name;
- params.interfaces;
- params.groups;
- params.templates;
- params.tags;
- id.

13.8. State-файл обработки

Микросервис должен сохранять последний успешно обработанный объект в файл.

Development state file:

state/cmdbkafka2zabbix-state.dev.json

Production/base state file:

state/cmdbkafka2zabbix-state.json

State должен содержать:
- last entityId;
- last eventType;
- input topic;
- input partition;
- input offset;
- output topic, если была публикация;
- признак outputPublished;
- skipReason, если сообщение было пропущено;
- processedAt.

При старте микросервис должен читать state-файл и писать в лог последний обработанный объект.
State-файлы являются runtime-данными и не должны попадать в git.

13.9. Логирование cmdbkafka2zabbix

ELK-сервера пока нет.

Текущий временный режим:
- structured JSON logs через Kafka;
- отдельный Kafka topic для логов.

Development log topic:

cmdbkafka2zabbix.logs.dev

Production/base config log topic:

cmdbkafka2zabbix.logs

Конфиг должен содержать секцию ElkLogging по тому же принципу, что и cmdbwebhooks2kafka.

13.10. Обработка событий create/update/delete

Текущий rules-файл содержит eventRoutingRules для:
- create -> host.create;
- update -> host.update;
- delete -> host.delete.

Для update/delete Zabbix API требует zabbix hostid.
Если во входном событии есть поле zabbix_hostid, микросервис формирует прямой host.update или host.delete.
Если zabbix_hostid отсутствует, микросервис формирует fallback host.get по вычисленному имени host, чтобы следующий обработчик мог получить hostid и затем выполнить update/delete.
Для fallback update/delete в JSON-RPC request добавляется служебный объект cmdb2monitoring.
Этот объект не отправляется в Zabbix API и удаляется третьим сервисом перед фактическим вызовом Zabbix.

Для delete fallback metadata содержит:
- eventType;
- entityId;
- host;
- fallbackForMethod=host.delete.

Для update fallback metadata содержит:
- eventType;
- entityId;
- host;
- fallbackForMethod=host.update;
- fallbackUpdateParams с целевыми параметрами host.update без hostid.

zabbixrequests2api после host.get:
- для fallbackForMethod=host.delete берет найденный hostid и выполняет host.delete;
- для fallbackForMethod=host.update берет найденный hostid и interfaceid, дополняет fallbackUpdateParams и выполняет host.update.

Rules-файл содержит отдельные виды правил:
- eventRoutingRules;
- validationRegex для входных полей;
- groupSelectionRules;
- templateSelectionRules;
- interfaceSelectionRules;
- tagSelectionRules;
- normalization rules;
- T4 templates для host.create, host.update, host.delete, host.get.

Будущие задачи:
- определить, где хранить связь CMDBuild id/code -> Zabbix hostid;
- определить стратегию dead-letter topic для ошибок конвертации;
- добавить отдельный аудит повторных fallback-событий и политику дедупликации.

14. Микросервис zabbixrequests2api

14.1. Назначение

Микросервис zabbixrequests2api является третьим этапом пайплайна:

CMDBuild -> cmdbwebhooks2kafka -> cmdbuild.webhooks.* -> cmdbkafka2zabbix -> zabbix.host.requests.* -> zabbixrequests2api -> Zabbix API.

Сервис читает JSON-RPC payload из Kafka, валидирует его, проверяет связанные справочники в Zabbix, вызывает Zabbix API и публикует результат выполнения в Kafka response topic.

14.2. Kafka input/output

Development input topic:

zabbix.host.requests.dev

Production/base input topic:

zabbix.host.requests

Development output topic:

zabbix.host.responses.dev

Production/base output topic:

zabbix.host.responses

Микросервис не создает Kafka topics при старте.
Input/output/log topics должны быть созданы заранее средствами инфраструктуры.

Kafka offsets должны коммититься только после:
- публикации response/error сообщения в output topic;
- записи state-файла.

14.3. Zabbix API

Zabbix API endpoint задается через конфиг:

Zabbix.ApiEndpoint

Текущий dev endpoint:

http://localhost:8081/api_jsonrpc.php

Поддерживаемые режимы авторизации:
- None;
- Token;
- Login;
- LoginOrToken.

Dev login:
- user: Admin;
- password: zabbix.

Production token/password не должны храниться в git. Для реального окружения использовать переменные окружения, секрет-хранилище или local config, исключенный из git.

14.4. Обязательные поля JSON-RPC

Каждый входной объект должен содержать:
- jsonrpc=2.0;
- method;
- params;
- id.

Для host.create обязательны:
- params.host;
- params.interfaces с минимум одним интерфейсом;
- params.groups с минимум одной host group;
- params.templates с минимум одним template.

Для host.update обязательны:
- params.hostid.

Для host.delete обязательны:
- params массив с минимум одним hostid.

Для host.get в текущем pipeline обязателен:
- params.filter.host с минимум одним host name.

Tags не создаются отдельным справочником Zabbix. Они передаются напрямую в host.create/host.update в поле params.tags.
Для tags проверяется наличие непустого поля tag.

14.5. Проверка справочников Zabbix

Перед host.create и host.update сервис должен проверять:
- наличие всех params.groups[].groupid через hostgroup.get;
- наличие всех params.templates[].templateid через template.get;
- наличие template group у найденных templates через template.get с selectGroups.

Если справочник отсутствует, сервис не вызывает host.create/host.update, а публикует error response.

Коды ошибок:
- missing_host_group;
- missing_template;
- missing_template_group;
- invalid_zabbix_request;
- invalid_json;
- zabbix_api_error;
- zabbix_http_error.

14.6. Gentle processing

Сервис не должен отправлять объекты в Zabbix непрерывным потоком.

Между обработкой объектов должна быть настраиваемая задержка:

Processing.DelayBetweenObjectsMs

Значение по умолчанию:

2000 ms

Retry-настройки также задаются через конфиг:
- Processing.MaxRetryAttempts;
- Processing.RetryDelayMs.

14.7. Response topic contract

В output topic zabbix.host.responses.* сервис публикует JSON:

{
  "source": "zabbixrequests2api",
  "success": true|false,
  "method": "host.create|host.update|host.delete|host.get",
  "entityId": "<Kafka key>",
  "requestId": "<JSON-RPC id>",
  "host": "<host name>",
  "errorCode": "<код ошибки или null>",
  "errorMessage": "<текст ошибки или null>",
  "zabbixRequestSent": true|false,
  "processedAt": "<UTC timestamp>",
  "input": {
    "topic": "<input topic>",
    "partition": 0,
    "offset": 0,
    "key": "<input key>"
  },
  "missing": {
    "hostGroups": [],
    "templates": [],
    "templateGroups": []
  },
  "zabbixResponse": { "...": "raw Zabbix response если был вызов API" }
}

14.8. State-файл обработки

Development state file:

state/zabbixrequests2api-state.dev.json

Production/base state file:

state/zabbixrequests2api-state.json

State должен содержать:
- last entityId;
- last method;
- input topic;
- input partition;
- input offset;
- output topic;
- outputPublished;
- zabbixRequestSent;
- success;
- errorCode;
- processedAt.

14.9. Логирование zabbixrequests2api

ELK-сервера пока нет.

Текущий временный режим:
- structured JSON logs через Kafka;
- отдельный Kafka topic для логов.

Development log topic:

zabbixrequests2api.logs.dev

Production/base config log topic:

zabbixrequests2api.logs

Конфиг должен содержать секцию ElkLogging по тому же принципу, что и предыдущие микросервисы.

14.10. Шаблоны Zabbix

Шаблоны мониторинга не являются JSON-файлами проекта.
Шаблоны мониторинга являются объектами внутри Zabbix.

JSON используется:
- как формат rules-файла конвертации;
- как формат JSON-RPC запроса к Zabbix API.

В JSON-RPC host.create сервис передает только ссылки на уже существующие шаблоны Zabbix:

{
  "templates": [
    { "templateid": "10564" },
    { "templateid": "10001" }
  ]
}

На текущем dev-стенде подтверждены шаблоны:
- templateid=10001: Linux by Zabbix agent;
- templateid=10564: ICMP Ping.

Если нужный templateid отсутствует в Zabbix, zabbixrequests2api должен остановить обработку объекта до вызова host.create/host.update и опубликовать ошибку missing_template в response topic.

18. Frontend/BFF monitoring-ui-api

18.1. Назначение

monitoring-ui-api является Node.js компонентом для frontend и backend-for-frontend.

Frontend не должен обращаться напрямую к Kafka, CMDBuild, Zabbix или IdP-интеграционным endpoint.
Все обращения выполняются через monitoring-ui-api.

Первый реализованный срез:
- статический frontend dashboard;
- Node.js HTTP API без внешних npm-зависимостей;
- server-side session;
- login form для режима без IdP;
- read-only health dashboard по health endpoints микросервисов;
- rules view с загрузкой JSON, validate, dry-run и upload;
- Zabbix catalog sync;
- CMDBuild catalog sync;
- console для SAML2/IdP настроек.

18.2. Авторизация

Поддерживаются два режима требований:

IdP enabled:
- целевая модель через единый IdP;
- основной протокол SAML2;
- предполагается, что CMDBuild и Zabbix обслуживаются тем же IdP;
- настройки IdP вносятся через console после авторизации.

IdP disabled:
- пользователь вводит CMDBuild login/password;
- пользователь вводит Zabbix login/password или Zabbix API token;
- credentials не хранятся в браузере;
- credentials хранятся только в памяти server-side session;
- production secrets не попадают в git.

Первый срез не реализует полноценный SAML2 login flow.
Он фиксирует настройки IdP/SAML2 и подготавливает структуру API/UI для следующей итерации.

18.3. Rules upload

Frontend должен уметь загружать JSON rules-файл.

Поток:
- пользователь выбирает локальный JSON;
- frontend отправляет JSON в monitoring-ui-api;
- API валидирует JSON;
- API может выполнить dry-run на тестовом CMDB payload;
- после успешной валидации API сохраняет файл в configured rules path;
- git commit из UI отключен по умолчанию через Rules.AutoCommit=false.

Upload доступен только роли admin.
Dry-run и validate могут быть доступны оператору в будущей RBAC-модели.

18.4. Zabbix catalog sync

monitoring-ui-api должен загружать из Zabbix:
- templates;
- host groups;
- template groups;
- known tags.

Catalog cache:

src/monitoring-ui-api/data/zabbix-catalog-cache.json

Cache не должен содержать секреты и не должен попадать в git.

Catalog используется для:
- UI-подсказок rules;
- validation rules upload;
- dry-run;
- отображения ID и человекочитаемых имен.

Runtime-проверки zabbixrequests2api остаются обязательными и не заменяются UI validation.

18.5. CMDBuild catalog sync

monitoring-ui-api должен загружать из CMDBuild:
- все классы объектов;
- active/inactive признак, если доступен;
- parent/prototype/superclass признаки, если доступны;
- attributes классов;
- lookup values, если доступны.

Catalog cache:

src/monitoring-ui-api/data/cmdbuild-catalog-cache.json

Cache не должен содержать секреты и не должен попадать в git.
