Техническое задание
Проект: cmdb2monitoring
Версия: 0.8.3
Дата: 2026-05-09

1. Назначение проекта

Проект cmdb2monitoring предназначен для интеграции CMDBuild с внешними системами мониторинга и обработки событий.
Разработка ведется в одном репозитории с разделением по микросервисам.

Первый микросервис: cmdbwebhooks2kafka.
Второй микросервис: cmdbkafka2zabbix.
Третий микросервис: zabbixrequests2api.
Четвертый микросервис: zabbixbindings2cmdbuild.
Frontend/BFF: monitoring-ui-api.

Назначение cmdbwebhooks2kafka:
- принимать webhook-события из CMDBuild;
- нормализовать входные данные;
- различать события create, update, delete;
- публиковать события в Kafka;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение cmdbkafka2zabbix:
- читать нормализованные события CMDBuild из Kafka;
- читать правила конвертации из Git-managed JSON-файла;
- применять регулярные выражения для валидации и выбора групп, шаблонов и интерфейсов Zabbix;
- формировать JSON-RPC payload, принимаемый Zabbix API;
- публиковать сформированный Zabbix JSON-RPC payload в отдельный Kafka topic;
- сохранять идентификатор последнего обработанного объекта в локальный state-файл;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение zabbixrequests2api:
- читать JSON-RPC запросы к Zabbix из Kafka;
- валидировать обязательные поля Zabbix payload;
- проверять наличие host groups, templates и template groups в Zabbix;
- выполнять вызовы Zabbix API;
- публиковать результат или ошибку в отдельный Kafka response topic;
- после успешного host.create/update/delete публиковать binding event для записи связи CMDBuild card/profile -> Zabbix hostid;
- обрабатывать объекты "gentle" режимом с настраиваемой задержкой между объектами;
- сохранять идентификатор последнего обработанного объекта в локальный state-файл;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение zabbixbindings2cmdbuild:
- читать binding events из Kafka;
- записывать `zabbix_main_hostid` в исходную карточку CMDBuild для основного host profile;
- создавать или обновлять карточки служебного класса `ZabbixHostBinding` для дополнительных host profiles;
- сохранять идентификатор последнего обработанного binding event в локальный state-файл;
- вести структурированные логи в Kafka-топик для будущей интеграции с ELK.

Назначение monitoring-ui-api:
- предоставлять Node.js frontend и backend-for-frontend для оператора интеграции;
- отображать health/status микросервисов;
- принимать login/password для CMDBuild и Zabbix на server-side session; для Zabbix предварительно использовать постоянный API key, если он задан;
- выполнять внешний login flow через MS AD LDAP/LDAPS или IdP SAML2/OAuth2/OIDC;
- хранить credentials только в server-side session;
- предоставлять admin-подменю `Авторизация` с выбором режима `Локальная`/`MS AD`/`IdP`, настройками SAML2/OAuth2/LDAP и таблицей соответствия ролей группам;
- загружать и валидировать JSON rules-файл;
- выполнять dry-run правил конвертации;
- визуализировать управление правилами конвертации CMDBuild -> rules -> Zabbix;
- предоставлять раздел `Настройка webhooks` для ролей `editor` и `admin`: читать текущие CMDBuild webhooks, анализировать rules, предлагать create/update/delete и применять выбранные операции в CMDBuild;
- посылать авторизованный сигнал `cmdbkafka2zabbix` на перечитывание правил конвертации;
- поддерживать session draft редактирование правил конвертации: добавление/удаление rules, undo/redo и save-as;
- синхронизировать catalog из Zabbix: templates, host groups, template groups, known tags;
- синхронизировать catalog из CMDBuild: классы, атрибуты, lookup-значения;
- хранить catalog cache локально без секретов.

2. Технологический стек

- Язык: C#.
- Платформа: .NET 10, target framework `net10.0`.
- Тип сервиса: ASP.NET Core Web API / Minimal API.
- Брокер сообщений: Apache Kafka.
- CMDB: CMDBuild.
- Логи: structured JSON logs, временно через Kafka-топик, в будущем через ELK.
- Репозиторий: monorepo.
- Frontend/BFF: Node.js >= 22 без прямого доступа браузера к Kafka/CMDBuild/Zabbix.

4. Микросервис cmdbwebhooks2kafka

4.1. HTTP endpoint

Микросервис должен принимать POST-запросы от CMDBuild:

POST /webhooks/cmdbuild

Health endpoint:

GET /health

4.2. Входной формат webhook

CMDBuild должен отправлять JSON body с плоской структурой:

Базовый контракт не фиксирует конкретные имена классов CMDBuild и пользовательских атрибутов. Обязательны только поля маршрутизации и идентификации события, а остальные source keys настраиваются в webhook body и `rules.source.fields`.

Минимальный контракт:

{
  "source": "cmdbuild",
  "eventType": "create|update|delete",
  "cmdbuildEvent": "card_create_after|card_update_after|card_delete_after",
  "className": "<CMDBuild class name>",
  "id": "{server:Id}",
  "code": "{server:Code}"
}

В примерах `{server:...}` обозначает выражение CMDBuild webhook для текущей dev-настройки; имя `server` не является требованием продукта и заменяется на фактическую переменную/класс, доступную в конкретной CMDBuild webhook-конфигурации.

Пример для текущей dev-модели, где в CMDBuild класс называется `Server`, lookup-атрибуты называются `OS` и `zabbixTag`, два дополнительных интерфейса называются `iLo`/`iLo2`, а два адреса управления называются `mgmt`/`mgmt2`:

{
  "source": "cmdbuild",
  "eventType": "create|update|delete",
  "cmdbuildEvent": "card_create_after|card_update_after|card_delete_after",
  "className": "Server",
  "id": "{server:Id}",
  "code": "{server:Code}",
  "ip_address": "{server:IPAddress}",
  "interface": "{server:iLo}",
  "interface2": "{server:iLo2}",
  "profile": "{server:mgmt}",
  "profile2": "{server:mgmt2}",
  "description": "{server:Description}",
  "os": "{server:OS}",
  "zabbixTag": "{server:zabbixTag}"
}

Поля могут быть расширены без изменения базового контракта. Имена `Computer`, `Notebook`, `PC`, `Server`, `tk`, `zabbixTag`, `iLo`, `iLo2`, `mgmt`, `mgmt2`, `interface`, `interface2`, `profile` и `profile2` являются примерами конкретной dev-модели CMDBuild/Zabbix, а не ограничениями ПО.

ПО читает те source keys, которые объявлены в `rules.source.fields`. Например, если в конкретной CMDBuild-модели у класса `Server` два дополнительных интерфейса называются `iLo1` и `iLo2`, а два адреса управления называются `mgmnt` и `mgmt2`, оператор выбирает один из двух вариантов:
- оставить в webhook нейтральные source keys, например `interface/interface2/profile/profile2`, и связать их с реальными CMDBuild attributes через `source.fields[].cmdbAttribute` или `source.fields[].cmdbPath`;
- использовать source keys, совпадающие с именами CMDBuild attributes, например `iLo1/iLo2/mgmnt/mgmt2`, и указать эти ключи в `source.fields[].source`, `hostProfiles[].interfaces[].valueField`, regex/rules и T4.

Если `iLo1` и `iLo2` должны быть дополнительными interfaces того же Zabbix host, в rules добавляются два `source.fields` и два элемента `hostProfiles[].interfaces` внутри основного profile. Если `mgmnt` и `mgmt2` должны быть отдельными объектами мониторинга, в rules добавляются два отдельных `hostProfiles[]` с собственными `hostNameTemplate`, `visibleNameTemplate`, `interfaces[]`, selection rules и при необходимости `createOnUpdateWhenMissing=true`.
Webhook body остается плоским. Если выбранный атрибут является reference, CMDBuild body передает numeric id первого reference attribute, а полный путь до leaf хранится в `source.fields[].cmdbPath` rules. Пример: payload содержит `"АтрибутReference": "<numeric id>"`, rules содержит `cmdbPath: "Класс.АтрибутReference.АтрибутLeaf"`, converter поднимает leaf через CMDBuild REST.
Для CMDBuild domain-связей используется `cmdbPath` вида `Класс.{domain:СвязанныйКласс}.АтрибутLeaf`, где `Класс` - имя класса текущей карточки, `domain` - ключевое слово, `СвязанныйКласс` - класс второго конца связи, `АтрибутLeaf` - leaf attribute связанной карточки. Это покрывает N:N domains, у которых в карточке нет reference attribute: UI catalog sync должен читать detailed `/domains/{domain}`, чтобы получить `source`/`destination` каждого domain, а converter берет id текущей карточки, читает CMDBuild relations, проверяет принадлежность связи текущему class и target class, затем поднимает leaf по существующим правилам reference/lookup.
Если domain path не вернул leaf, converter обязан считать source field отсутствующим и не использовать id текущей карточки как значение leaf. `cmdbkafka2zabbix` не должен кэшировать CMDBuild relations между событиями, потому что relations могут быть созданы после `card_create_after`; последующий `card_update_after` должен читать актуальные relations.
`cmdbkafka2zabbix` также не должен хранить runtime cache карточек CMDBuild и lookup-значений между событиями. При `card_update_after` исходной карточки lookup, reference leaf и domain leaf значения перечитываются через CMDBuild REST заново. Изменение только связанной карточки или domain-связи без webhook/event исходной карточки не инициирует пересчет мониторинга.
Максимальная глубина рекурсии для `domain`/`reference`/`lookup` путей задается в `Runtime-настройки` параметром `Максимальная глубина рекурсии domains&reference&lookups`. Допустимый диапазон `2..5`, значение по умолчанию `2`. После изменения параметра оператор должен выполнить logout и пересинхронизацию CMDBuild catalog; до этого UI может продолжать использовать глубину из текущей сессии или старого cache. Новые source fields с `cmdbPath` должны получать `resolve.maxDepth`, соответствующий активной глубине catalog sync.
Leaf-состояние связанной domain-карточки не должно автоматически останавливать мониторинг исходного объекта. Если путь вида `Класс.{domain:СвязанныйКласс}.АтрибутСостоянияСвязанногоEndpoint` возвращает `do_not_monitor`, это означает только запрет использовать соответствующий leaf endpoint как Zabbix interface или отдельный hostProfile. Исходный объект можно продолжать ставить на мониторинг по другим разрешенным адресам. В таком случае поведение формулируется как "не использовать связанный endpoint", а не как "не ставить на мониторинг" исходный объект и не как "остановить мониторинг объекта".
Если по атрибутам самого экземпляра нужно принять решение "не ставить на мониторинг", это не считается ошибкой заполнения webhook. Оператор добавляет source field, например `monitoringPolicy`, и правило `monitoringSuppressionRules`. При совпадении условия на `create/update` converter должен вернуть skip reason `monitoring_suppressed:*` и не публиковать Zabbix request. Для `delete` suppression не применяется: удаление может быть сигналом "остановить мониторинг объекта", поэтому ранее созданный host должен иметь возможность пройти cleanup/fallback обработку.
После переименования additional profiles новые Zabbix hosts получают suffix `-profile`/`-profile2`; ранее созданные hosts со старыми suffix не переименовываются автоматически.

Для приведенного dev-примера CMDBuild корректно подставляет:
- {server:Id} в поле id;
- {server:Code} в поле code;
- {server:IPAddress} в поле ip_address для основного Server host;
- {server:iLo} в поле interface для дополнительного интерфейса основного Server host;
- {server:iLo2} в поле interface2 для дополнительного интерфейса основного Server host;
- {server:mgmt} в поле profile для отдельного дополнительного Server monitoring object;
- {server:mgmt2} в поле profile2 для отдельного дополнительного Server monitoring object;
- {server:OS} в поле os;
- {server:zabbixTag} в поле zabbixTag.

Особенность CMDBuild lookup-полей:
- для lookup webhook обычно передает внутренний numeric id, а не текстовое значение;
- в текущем dev-примере это относится к lookup `OS` и `zabbixTag`;
- rules-файл второго микросервиса маппит эти id в смысловые значения для Zabbix через `type=lookup`, `lookupType` и `resolve.valueMode=code`.

Для reference и lookup leaf поддерживается итеративный путь в rules:
- `Класс.АтрибутReference.АтрибутScalar`: webhook передает numeric id reference, converter читает linked card и берет scalar leaf;
- `Класс.АтрибутReference1.АтрибутReference2.АтрибутScalar`: converter дважды проходит reference и берет финальный scalar;
- `Класс.АтрибутReference1.АтрибутReference2.АтрибутLookup`: converter проходит reference-цепочку и преобразует финальный lookup id в code;
- если leaf сам является lookup, `resolve.leafType=lookup` и `resolve.lookupType` задают справочник.

4.3. Поддерживаемые события CMDBuild

CMDBuild webhook events:
- card_create_after -> eventType=create;
- card_update_after -> eventType=update;
- card_delete_after -> eventType=delete.

4.4. Классы CMDBuild и границы конкретной модели

ПО не содержит встроенного списка классов CMDBuild. Поддерживаемые классы задаются конфигурацией CMDBuild webhook и rules-файлом:
- `source.entityClasses`;
- условиями по `className`;
- `source.fields`;
- `hostProfiles[]`;
- selection rules;
- T4 templates.

Текущая dev-модель использует базовый класс CMDBuild `Computer`.

Так как в этой dev-модели `Computer` является prototype-классом, webhook настраивается на конкретные наследники:
- Notebook;
- PC;
- Server;
- tk.

Для каждого класса должны быть заведены webhook-записи на create, update, delete.

В dev-модели для подклассов `Computer` используются дополнительные атрибуты:
- OS: lookup со значениями Windows, Linux, Windows server, vmware;
- zabbixTag: lookup со значениями tag1, tag2, tag3.

В другой CMDBuild-модели классы и атрибуты могут называться иначе. Подключение конкретного класса, например `Network device`, допускается без изменения кода, если:
- класс существует в CMDBuild catalog;
- для класса заведены webhook-записи на нужные события;
- webhook body передает обязательные поля `id`, `code`, `className` и либо IP, либо DNS-атрибут для создания/обновления Zabbix host;
- новые атрибуты добавлены в rules `source.fields` и участвуют в правилах/T4 без изменения базового контракта.

4.5. Заголовки webhook из CMDBuild

CMDBuild webhook должен передавать только:

Authorization: Bearer XXXXX

Content-Type вручную в CMDBuild webhook headers не указывать.
Причина: CMDBuild сам добавляет Content-Type. Если прописать его вручную, запрос приходит как application/json,application/json.

Микросервис должен быть устойчив к некорректному/дублированному Content-Type и читать JSON body вручную.

5. Авторизация webhook

Требуется Bearer token авторизация входящих webhook-запросов.

Режимы токена:
- static: токен хранится в конфиге микросервиса;
- disabled: только для окружений, где входящий webhook защищен внешним gateway/reverse proxy;
- dynamic: заглушка под будущую реализацию, источник динамического токена пока не определен.

Реализация static mode:
- `CmdbWebhook:AuthorizationMode=Static`;
- `CmdbWebhook:BearerToken` или env `CmdbWebhook__BearerToken`;
- при отсутствующем или неверном `Authorization: Bearer ...` сервис должен вернуть `401` и не публиковать событие в Kafka.

На стороне CMDBuild токен прописывается вручную в webhook header Authorization.

Важно: секретное значение токена не должно храниться в git в production-конфиге. Для реального окружения использовать переменные окружения, секрет-хранилище или local config, исключенный из git.

6. Kafka

6.1. Топик бизнес-событий

Микросервис публикует нормализованные CMDBuild webhook-события в Kafka.

Development topic:

cmdbuild.webhooks.dev

Production/base config topic:

cmdbuild.webhooks

6.2. Текущий режим подключения Kafka

Текущая локальная Kafka работает без авторизации:
- SecurityProtocol: Plaintext;
- SASL отключен;
- TLS отключен;
- username/password не используются.

Адреса Kafka:
- с хоста: localhost:9092 или 127.0.0.1:9092;
- из Docker-сети: kafka:29092.

6.3. Kafka security config

Все параметры подключения должны задаваться через конфиг:
- BootstrapServers;
- Topic;
- ClientId;
- SecurityProtocol;
- SaslMechanism;
- Username;
- Password;
- Acks;
- EnableIdempotence;
- MessageTimeoutMs.

Сейчас используется Plaintext. В будущем должна быть возможность включить SASL/TLS через конфиг без изменения кода.

6.4. Kafka message contract

Микросервис публикует envelope:

{
  "source": "cmdbuild",
  "eventType": "create|update|delete",
  "entityType": "<CMDBuild class name>",
  "entityId": "<id или code>",
  "receivedAt": "<UTC timestamp>",
  "payload": { ... исходный webhook payload ... }
}

Правило entityId:
- сначала читать поле id;
- затем code;
- затем entityId.

Kafka key:
- entityId, если найден;
- иначе eventType.

6.5. Kafka topics

Микросервис не создает Kafka topics при старте.
Необходимые topics должны быть созданы заранее средствами инфраструктуры.

Обязательные topics:

- cmdbwebhooks2kafka.logs.dev для dev-логов;
- cmdbwebhooks2kafka.logs для production/base config логов.

7. Логирование и ELK

ELK-сервера пока нет.

Требуется заложить настройки и код так, чтобы позже подключить ELK без изменения бизнес-логики.

Текущий временный режим:

- формат логов: JSON;
- отдельный Kafka-топик для логов.

Development log topic:

cmdbwebhooks2kafka.logs.dev

Production/base config log topic:

cmdbwebhooks2kafka.logs

Конфиг должен содержать секцию ElkLogging:
- Enabled;
- Mode;
- Elk.Endpoint;
- Elk.Index;
- Elk.ApiKey;
- Kafka настройки для лог-топика;
- MinimumLevel;
- ServiceName;
- Environment;

Конфиг каждого .NET-микросервиса должен содержать секцию DebugLogging:

Поддержанные значения Level:
- Basic;
- Verbose.

DebugLogging выключен по умолчанию. При включении debug-события пишутся через обычный ILogger на уровне Information, чтобы попадать в Docker stdout/stderr, Kafka log topic при включенном Kafka logging, ELK при включенном ELK sink и syslog при соответствующей настройке Docker logging driver. Level Basic фиксирует ключевые этапы обработки события, Level Verbose дополнительно раскрывает payload/request/response JSON и предназначен только для временной диагностики.

13. Микросервис cmdbkafka2zabbix

13.1. Назначение

Микросервис cmdbkafka2zabbix является вторым этапом пайплайна:

CMDBuild -> cmdbwebhooks2kafka -> Kafka topic cmdbuild.webhooks.* -> cmdbkafka2zabbix -> Kafka topic zabbix.host.requests.*.

Микросервис не вызывает Zabbix API напрямую на текущем этапе.
Он формирует JSON-RPC request, совместимый с Zabbix host.create, и публикует его в Kafka.

13.2. Kafka input

Development input topic:

cmdbuild.webhooks.dev

Production/base input topic:

cmdbuild.webhooks

Consumer group должен задаваться через конфиг.
Kafka offsets должны коммититься только после успешной обработки сообщения или после осознанного skip некорректного/неподдерживаемого события.

13.3. Kafka output

Development output topic:

zabbix.host.requests.dev

Production/base output topic:

zabbix.host.requests

В output topic публикуется тело сообщения в виде JSON-RPC payload для Zabbix, например method=host.create.
Kafka message key:
- entityId, если он есть;
- иначе имя Zabbix host;
- иначе eventType.

Микросервис не создает Kafka topics при старте.
Input/output/log topics должны быть созданы заранее средствами инфраструктуры.

13.4. Правила конвертации из Git

Правила конвертации хранятся в JSON-файле:

rules/cmdbuild-to-zabbix-host-create.json

Источник правил настраивается через конфиг:
- RepositoryPath;
- ReadFromGit;
- RepositoryUrl;
- RulesFilePath;
- PullOnStartup;
- PullOnReload;
- GitExecutablePath;
- TemplateName.

В dev-режиме RepositoryPath указывает на текущий monorepo.
В production RepositoryPath должен указывать на локальную Git working copy с правилами. Clone/fetch/commit/push этой working copy выполняются вне `monitoring-ui-api`; микросервис только делает pull/read по настройке. Если `ReadFromGit=false`, сервис читает только локальный файл и не выполняет git-команды. Если `ReadFromGit=true`, `RepositoryUrl` хранит URL источника для оператора/логов, например `https://git.example.org/cmdb2monitoring/conversion-rules.git`, а `PullOnStartup`/`PullOnReload` управляют выполнением `git pull --ff-only` в уже подготовленной working copy.
Для нашей dev/test системы режим по умолчанию - чтение с диска из файла `rules/cmdbuild-to-zabbix-host-create.json`. При включении `ReadFromGit=true` внутри repository ожидается этот же файл по пути `RulesFilePath`; если оператор меняет `RulesFilePath`, ожидаемый путь внутри repository меняется вместе с ним.
Rules-файл обязан содержать `schemaVersion` для совместимости формата и `rulesVersion` для визуального контроля редакции правил. `rulesVersion` должен включать дату и время изменения в человекочитаемом виде, например `2026.05.08-142503-serveri-webhook-fix`.
UI при `Save file as` и `Сохранить в git` должен формировать новый `rulesVersion`: обновлять префикс даты/времени до секунд и сохранять смысловой suffix предыдущей версии, например `test15-profile-ip-leaf-fix`.

13.4.1. Сигнал перечитывания правил

`cmdbkafka2zabbix` должен поддерживать management endpoint `POST /admin/reload-rules`.
Endpoint должен:
- требовать `Authorization: Bearer <Service:RulesReloadToken>`;
- вызывать абстракцию `IConversionRulesProvider.ReloadAsync`, а не выполнять Git-операции напрямую в endpoint;
- для текущего provider при `ConversionRules:ReadFromGit=true` и `ConversionRules:PullOnReload=true` выполнять `git pull --ff-only`, затем перечитывать rules-файл;
- при `ReadFromGit=false` или `PullOnReload=false` только сбрасывать cache и перечитывать локальный rules-файл;
- возвращать имя rules, schemaVersion, rulesVersion, location, version/commit и reloadedAt.

`cmdbkafka2zabbix` должен поддерживать read-only management endpoint `GET /admin/rules-status`, который возвращает имя rules, `schemaVersion`, `rulesVersion`, location, version/commit, признак чтения из git и время проверки без принудительного reload.
`monitoring-ui-api` в Панели рядом с кнопкой `Перечитать правила конвертации` должен показывать две версии: версию правил, загруженную на микросервисе, и версию rules-файла, который сейчас читает система управления.
После успешного нажатия `Перечитать правила конвертации` UI должен заново прочитать `/api/rules/current` и обновить данные rules-файла на Панели и в связанных индикаторах источника; состояние "готово" не должно оставлять старые данные файла в уже открытом интерфейсе.

`monitoring-ui-api` должен показывать кнопку `Перечитать правила конвертации` в карточке `cmdbkafka2zabbix` на панели для ролей `editor` и `admin`.
При будущей смене места хранения правил меняется provider/config, но не HTTP-контракт и не поведение UI.

13.5. Данные для создания host в Zabbix

Для Zabbix host.create должны быть сформированы:
- host: техническое имя хоста;
- name: отображаемое имя;
- interfaces: IP, тип интерфейса, порт;
- groups: массив groupid;
- templates: массив templateid;
- tags: массив tag/value;
- status;
- inventory_mode.

Zabbix API принимает groupid/templateid, поэтому человекочитаемые имена групп и шаблонов в rules-файле должны храниться вместе с реальными ID из Zabbix.

Локально подтвержден Zabbix:
- URL UI: http://localhost:8081;
- API endpoint: http://localhost:8081/api_jsonrpc.php;
- версия: 7.0.25;
- dev user/password: Admin/zabbix.

Используемые локальные справочники:
- group Linux servers: groupid=2;
- group Discovered hosts: groupid=5;
- group Virtual machines: groupid=6;
- group Hypervisors: groupid=7;
- group Applications: groupid=19;
- group Databases: groupid=20;
- template Linux by Zabbix agent: templateid=10001;
- template Windows by Zabbix agent: templateid=10081;
- template ICMP Ping: templateid=10564;
- template Generic by SNMP: templateid=10563;
- template Network Generic Device by SNMP: templateid=10226.

Используемые CMDBuild lookup id для dev-окружения:
- OS Windows: 105130;
- OS Linux: 105146;
- OS Windows server: 105152;
- OS vmware: 105161;
- zabbixTag tag1: 106857;
- zabbixTag tag2: 106852;
- zabbixTag tag3: 106850.

Rules-файл поддерживает как человекочитаемые значения, так и numeric lookup id, потому что webhook CMDBuild передает lookup-поля в виде id. В штатном потоке `cmdbkafka2zabbix` преобразует id в lookup `code` до применения правил, а numeric id остаются в regex как fallback для окружений без настроенного CMDBuild resolver.
В текущем dev-примере при создании host в Zabbix поля OS и zabbixTag преобразуются в Zabbix tags:
- cmdb.os=<Windows|Linux|Windows server|vmware>;
- cmdb.zabbixTag=<tag1|tag2|tag3>.

13.6. Регулярные выражения и выбор параметров

Regex используется не только для валидации, но и для выбора параметров Zabbix.

В rules-файле должны быть отдельные блоки:
- validationRegex для проверки входных данных, например ip_address;
- groupSelectionRules для выбора групп Zabbix;
- templateSelectionRules для выбора шаблонов мониторинга;
- interfaceSelectionRules для выбора типа интерфейса;
- tagSelectionRules для выбора Zabbix tags;
- normalization.hostName.regexReplacements для нормализации имени host.

Правила применяются по priority.
Для групп и шаблонов допускается объединение нескольких совпавших правил с дедупликацией по groupid/templateid.
Для `tagSelectionRules` и `groupSelectionRules` допускается явный динамический режим `targetMode=dynamicFromLeaf`. В этом режиме rule обязан содержать `valueField`; для tags формируется `tags[]` из leaf-значения, а для host groups формируется `groups[]` с `name` и `createIfMissing` до этапа Zabbix writer. После resolve/create Zabbix writer обязан подставить найденный или созданный `groupid` в тот же `host.create`/`host.update` payload, чтобы текущий host был привязан к тем же host groups, которые пришли из leaf. Динамический режим разрешен только для Tags и Host groups и не применяется к templates, interfaces, inventory и macros.
После выбора шаблонов должны применяться `templateConflictRules`: если выбран шаблон, уже содержащий конфликтующий item key или конфликтующую inventory-привязку, rules могут удалить другой templateid из итогового payload. В текущем dev-наборе `ICMP Ping` и agent-шаблоны удаляются при выборе `HP iLO by SNMP` или `Generic by SNMP`, потому что эти SNMP-шаблоны уже содержат item key `icmpping` и заполняют inventory field `Name`.
Для update fallback через `host.get` конвертер должен передавать `templates_clear`, а Zabbix writer должен применять его только к templateid, которые реально привязаны к host по данным `selectParentTemplates`.
При `host.update` Zabbix writer должен сохранять внешние назначения, которые не управляются текущими rules: `groups[]` объединяются по `groupid`, `templates[]` объединяются по `templateid` с учетом явного удаления через `templates_clear`, `tags[]` объединяются по паре `tag/value`, `macros[]` объединяются по `macro`, а `inventory` объединяется по имени поля. Значения из rules имеют приоритет над одноименными текущими значениями. `interfaces[]` не являются merge-полем: их итоговый состав задается rules, а writer только переносит существующие `interfaceid`.
Пример несовместимости: host с `Windows by Zabbix agent` не может без очистки получить `HP iLO by SNMP`, потому что оба шаблона пытаются заполнить inventory field `Name`. В этом случае rules должны выбрать целевой SNMP-шаблон и передать конфликтующий agent-шаблон в `templates_clear` для update fallback.
Для интерфейса выбирается первое совпавшее правило, иначе fallback.

13.6.1. Host profiles и несколько интерфейсов

Rules-файл должен поддерживать два режима обработки объектов с несколькими адресами:

1. Один CMDB object -> один Zabbix host с несколькими interfaces[].
   Применяется, если основной IP и дополнительные interface IP относятся к одному объекту мониторинга.
   В этом случае несколько элементов описываются внутри одного hostProfiles[].interfaces.

2. Один CMDB object -> несколько Zabbix hosts.
   Применяется, если основной сервер и дополнительный profile object должны иметь разные host names, templates, groups, tags или lifecycle.
   В этом случае rules содержит несколько hostProfiles[], например main, profile и profile2.

Host profile должен содержать:
- name;
- when condition;
- hostNameTemplate;
- visibleNameTemplate;
- interfaces[] с interfaceProfileRef, mode и valueField.

interfaceProfileRef выбирает профиль Zabbix interfaces[] из defaults.interfaceProfiles. Профиль задает тип мониторинга agent/SNMP/IPMI/JMX, порт, main/useip и SNMP details; это не отдельный объект Zabbix.
mode=ip заполняет interfaces[].ip и useip=1.
mode=dns заполняет interfaces[].dns и useip=0.
valueField ссылается на поле из `source.fields`; имя поля выбирает оператор в rules. Примеры: `ipAddress`, `dnsName`, `interface1IpAddress`, `interface2IpAddress`, `mgmntIpAddress`, `mgmt2IpAddress`.
Редактор правил должен блокировать явную ошибку semantic mode: IP-looking field нельзя использовать как `mode=dns`, DNS/FQDN-looking field нельзя использовать как `mode=ip`.

Имена классов, attributes и source fields не зашиты в код. Ниже приведен пример конкретной модели: если класс CMDBuild называется `Server`, два дополнительных интерфейса называются `iLo1` и `iLo2`, а два адреса управления `mgmnt` и `mgmt2` нужно вынести в отдельные Zabbix hosts, тогда rules настраиваются так:
- `source.fields` описывает основной IP и четыре дополнительных поля, например `interface1IpAddress.source=iLo1`, `interface2IpAddress.source=iLo2`, `mgmntIpAddress.source=mgmnt`, `mgmt2IpAddress.source=mgmt2`;
- основной hostProfile `main` содержит agent interface по основному IP и два дополнительных SNMP interfaces с `valueField=interface1IpAddress/interface2IpAddress`;
- отдельные hostProfiles, например `mgmnt` и `mgmt2`, содержат собственные `hostNameTemplate`, `visibleNameTemplate`, `interfaces[]` и selection rules для отдельных объектов мониторинга;
- если отдельный profile должен создаваться при первом появлении адреса на update, для него задается `createOnUpdateWhenMissing=true`;
- если source field является reference, lookup или domain path, путь до leaf задается через `source.fields[].cmdbPath`, а lookup преобразуется через `resolve`; пример domain-связи без attribute в карточке: `Класс.{domain:СвязанныйКласс}.АтрибутScalar`.
- если source field используется как признак исключения из мониторинга самого объекта, он участвует в `monitoringSuppressionRules`; пример: `Класс.АтрибутMonitoringPolicy = do_not_monitor` означает "не ставить на мониторинг" для `create/update`, но не блокирует `delete`, потому что `delete` может быть командой "остановить мониторинг объекта";
- если `do_not_monitor` находится не на самом объекте, а на leaf-состоянии связанной domain-карточки, например `Класс.{domain:СвязанныйКласс}.АтрибутСостоянияАдреса = do_not_monitor`, это запрещает использовать только соответствующий leaf endpoint. Основной объект продолжает обрабатываться по другим разрешенным адресам.

Текущая dev-схема является частным случаем этого подхода:
- один основной hostProfile `main`: `ip_address -> ipAddress` как основной agent interface, `interface -> interfaceIpAddress` и `interface2 -> interface2IpAddress` как два дополнительных SNMP interface в том же Zabbix host;
- отдельный hostProfile `profile`: `profile -> profileIpAddress`, Zabbix host suffix `-profile`, SNMP interface type 2, port 161;
- отдельный hostProfile `profile2`: `profile2 -> profile2IpAddress`, Zabbix host suffix `-profile2`, SNMP interface type 2, port 161;
- `main` с `interface/interface2` использует шаблон HP iLO by SNMP, а `profile/profile2` используют отдельные SNMP monitoring rules; конфликтующие ICMP Ping и agent-шаблоны удаляются rules-блоком `templateConflictRules`, если их выбрали другие совпавшие rules.

Ограничения по количеству IP:
- микросервис cmdbkafka2zabbix не должен вводить собственный жесткий числовой лимит на количество `interfaces[]`; количество interfaces определяется rules `hostProfiles[].interfaces`;
- конкретная поставка ограничена явно описанными fields в webhook/rules; текущая dev-схема как пример использует `ip_address`, `interface`, `interface2`, `profile` и `profile2`, но эти имена можно заменить или расширить в rules;
- Zabbix допускает несколько interfaces у одного host, но для каждого interface type должен быть только один основной interface `main=1`; дополнительные interfaces того же type должны быть `main=0`;
- добавление нового фиксированного IP в основной Zabbix host требует добавить CMDBuild attribute, webhook field, `source.fields` и элемент `hostProfiles[].interfaces`; для второго и последующих interfaces того же Zabbix type необходимо задать `main=0`;
- добавление нового отдельного monitoring profile требует добавить named source field и отдельный `hostProfile` с собственным suffix, например `profile3`; в UI это выполняется через отдельный блок `Профили мониторинга`, после чего templates/groups/tags назначаются отдельными rules через виртуальное поле `hostProfile`; если объект должен создаваться при первом появлении адреса на update, profile должен включать `createOnUpdateWhenMissing=true`;
- произвольное количество IP из одного webhook-поля в формате массива текущим контрактом не поддерживается. Для этого потребуется отдельное развитие модели webhook/rules/T4 с массивом interfaces или profiles и итерацией по нему.

Поведение при частично заполненных дополнительных адресах:
- если в карточке Server заполнен только основной `ip_address`, должен создаваться/обновляться только основной hostProfile `main` с одним agent interface;
- пустые `interface` и `interface2` не считаются ошибкой и не добавляются в `interfaces[]`; при последующем update с заполненными `interface`/`interface2` основной `main` host должен получить update payload с дополнительными SNMP interfaces;
- пустые `profile` и `profile2` на create/update не выбирают соответствующие дополнительные hostProfiles, поэтому отдельные Zabbix hosts `-profile` и `-profile2` не создаются;
- если `profile` или `profile2` впервые заполнен только на update, для profiles с `createOnUpdateWhenMissing=true` должен выполняться upsert: `zabbixrequests2api` делает `host.get`, при найденном host выполняет `host.update`, а при отсутствующем host валидирует `fallbackCreateParams` и выполняет `host.create`;
- если `profile` или `profile2` очищен после создания дополнительного Zabbix host, автоматическое удаление не выполняется: оператор или внешний процесс должен самостоятельно принять решение по объекту мониторинга Zabbix, например оставить, отключить или удалить его;
- для delete profiles `profile` и `profile2` обрабатываются даже при пустых адресах, чтобы удалить ранее созданные дополнительные hosts по вычисленному имени; отсутствие такого host фиксируется как `host_not_found`.

При нескольких hostProfiles микросервис cmdbkafka2zabbix публикует несколько сообщений в zabbix.host.requests.* для одного входного CMDB event.
State входного offset должен записываться только после публикации всех сообщений по этому событию.

Для update/delete без zabbix_hostid каждый host profile должен проходить через fallback host.get по своему вычисленному host name.
В request metadata должно передаваться hostProfile, чтобы логи и responses позволяли понять, какой profile обрабатывался.

Текущие правила выбора по OS:
- Linux или lookup id 105146 -> группа Linux servers, шаблон Linux by Zabbix agent, tag cmdb.os=Linux;
- Windows или lookup id 105130 -> группа Discovered hosts, шаблон Windows by Zabbix agent, tag cmdb.os=Windows;
- Windows server или lookup id 105152 -> группа Discovered hosts, шаблон Windows by Zabbix agent, tag cmdb.os=Windows server;

13.7. T4-шаблон

Формирование итогового JSON-RPC payload должно выполняться полноценным T4 engine из rules-файла.

Используемый runtime engine:

Mono.TextTemplating

Текущий шаблон:

t4Templates.hostCreateJsonRpcRequestLines

Настройки T4 engine задаются в конфиге микросервиса:
- ConversionRules.TemplateEngine;
- ConversionRules.TemplateName;
- ConversionRules.AddDefaultDirectives.

Если шаблон в rules-файле не содержит собственную директиву <#@ template ... #>, сервис добавляет стандартные directives автоматически:
- template language=C#;
- assembly reference на сборку микросервиса;
- import namespace CmdbKafka2Zabbix.Conversion;
- parameter Model типа CmdbKafka2Zabbix.Conversion.ZabbixHostCreateModel.

Если в будущем rules-файл будет содержать полностью самостоятельный T4-шаблон со своими directives, автоматическое добавление directives можно отключить через ConversionRules.AddDefaultDirectives=false.

Шаблон должен формировать JSON-RPC request с:
- jsonrpc=2.0;
- method=host.create;
- params.host;
- params.name;
- params.interfaces из Model.Interfaces;
- params.groups;
- params.templates;
- params.tags;
- id.

Для обратной совместимости Model.Interface остается первым элементом Model.Interfaces, но новые шаблоны должны использовать Model.Interfaces.

13.10. Обработка событий create/update/delete

Текущий rules-файл содержит eventRoutingRules для:
- create -> host.create;
- update -> host.update;
- delete -> host.delete.

Для update/delete Zabbix API требует zabbix hostid.
Если во входном событии есть поле zabbix_hostid, микросервис формирует прямой host.update или host.delete.
Если zabbix_hostid отсутствует, микросервис должен сначала попытаться прочитать сохраненную связь из CMDBuild:
- для основного host profile прочитать исходную карточку и атрибут `Cmdbuild:MainHostIdAttributeName`, default `zabbix_main_hostid`;
- для дополнительного host profile прочитать класс `Cmdbuild:BindingClassName`, default `ZabbixHostBinding`, и найти активную карточку по `OwnerClass + OwnerCardId + HostProfile`.
Если сохраненная связь отсутствует или CMDBuild resolver недоступен, микросервис формирует fallback host.get по вычисленному имени host, чтобы следующий обработчик мог получить hostid и затем выполнить update/delete.
Для fallback update/delete в JSON-RPC request добавляется служебный объект cmdb2monitoring.
Этот объект не отправляется в Zabbix API и удаляется третьим сервисом перед фактическим вызовом Zabbix.

Источник zabbix_hostid:
- поле zabbix_hostid/zabbixHostId не создается системой автоматически во входном CMDBuild webhook;
- поле может прийти из отдельного CMDBuild-атрибута, если инсталляция сама хранит Zabbix hostid в карточке;
- поле может быть добавлено внешним enrichment-процессом между CMDBuild webhook и cmdbkafka2zabbix;
- поле может быть прочитано rules как source.fields.zabbixHostId, если оно явно сопоставлено с payload;
- поле может быть прочитано автоматически из `zabbix_main_hostid` или `ZabbixHostBinding`, если оно уже записано сервисом `zabbixbindings2cmdbuild`.

Штатная идентификация host при update/delete без zabbix_hostid:
- cmdbkafka2zabbix вычисляет technical host name по rules normalization.hostName или hostProfiles[].hostNameTemplate;
- technical host name должен строиться из стабильной CMDBuild-идентичности: className, id, неизменяемый code, имя profile;
- IP/DNS не должны использоваться как часть идентификатора host, если допускается смена IP/DNS;
- zabbixrequests2api выполняет host.get по filter.host, берет найденный hostid и затем формирует host.update/host.delete;
- тег cmdb.id может записываться в Zabbix для аудита и ручного поиска, но текущий fallback lookup не ищет host по tag;
- processing state файлы сервисов не являются реестром связей CMDBuild id -> Zabbix hostid.

Смена IP-адреса:
- host идентифицируется по technical host name, поэтому изменение IP не должно создавать новый Zabbix host;
- новый IP передается в fallbackUpdateParams.interfaces;
- при update zabbixrequests2api подставляет существующий interfaceid: сначала по точному совпадению type/useip/ip/dns/port, затем по type/port/address, а для первого interface при отсутствии совпадения берет первый существующий interfaceid;
- поэтому смена основного IP обновляет interface существующего host;
- для дополнительных interfaces без стабильного признака профиля возможна неоднозначность, поэтому правила должны фиксировать profile/order/type/port достаточно явно.

Для delete fallback metadata содержит:
- eventType;
- entityId;
- host;
- fallbackForMethod=host.delete.

Для update fallback metadata содержит:
- eventType;
- entityId;
- host;
- fallbackForMethod=host.update;
- fallbackUpdateParams с целевыми параметрами host.update без hostid.

zabbixrequests2api после host.get:
- для fallbackForMethod=host.delete берет найденный hostid и выполняет host.delete;
- для fallbackForMethod=host.update берет найденный hostid и interfaceid, дополняет fallbackUpdateParams и выполняет host.update.

Rules-файл содержит отдельные виды правил:
- eventRoutingRules;
- validationRegex для входных полей;
- groupSelectionRules;
- templateSelectionRules;
- interfaceSelectionRules;
- tagSelectionRules;
- normalization rules;
- T4 templates для host.create, host.update, host.delete, host.get.

Будущие задачи:
- определить, где хранить связь CMDBuild id/code -> Zabbix hostid;
- определить стратегию dead-letter topic для ошибок конвертации;
- добавить отдельный аудит повторных fallback-событий и политику дедупликации.

14. Микросервис zabbixrequests2api

14.1. Назначение

Микросервис zabbixrequests2api является третьим этапом пайплайна:

CMDBuild -> cmdbwebhooks2kafka -> cmdbuild.webhooks.* -> cmdbkafka2zabbix -> zabbix.host.requests.* -> zabbixrequests2api -> Zabbix API.

Сервис читает JSON-RPC payload из Kafka, валидирует его, проверяет связанные справочники в Zabbix, вызывает Zabbix API и публикует результат выполнения в Kafka response topic.

14.2. Kafka input/output

Development input topic:

zabbix.host.requests.dev

Production/base input topic:

zabbix.host.requests

Development output topic:

zabbix.host.responses.dev

Production/base output topic:

zabbix.host.responses

Development binding output topic:

zabbix.host.bindings.dev

Production/base binding output topic:

zabbix.host.bindings

Микросервис не создает Kafka topics при старте.
Input/output/binding-output/log topics должны быть созданы заранее средствами инфраструктуры.

Kafka offsets должны коммититься только после:
- публикации response/error сообщения в output topic;
- записи state-файла.
Ошибка публикации binding event не должна приводить к повторной отправке уже выполненного запроса в Zabbix: сервис пишет ошибку в лог, но основной response и Kafka offset фиксируются.

14.3. Zabbix API

Zabbix API endpoint задается через конфиг:

Zabbix.ApiEndpoint

Текущий dev endpoint:

http://localhost:8081/api_jsonrpc.php

Поддерживаемые режимы авторизации:
- None;
- Token;
- Login;
- LoginOrToken.

Dev login:
- user: Admin;
- password: zabbix.

Production token/password не должны храниться в git. Для реального окружения использовать переменные окружения, секрет-хранилище или local config, исключенный из git.

14.4. Обязательные поля JSON-RPC

Каждый входной объект должен содержать:
- jsonrpc=2.0;
- method;
- params;
- id.

Для host.create обязательны:
- params.host;
- params.interfaces с минимум одним интерфейсом;
- params.groups с минимум одной host group;
- params.templates с минимум одним template.

Для host.update обязательны:
- params.hostid.

Для host.delete обязательны:
- params массив с минимум одним hostid.

Для host.get в текущем pipeline обязателен:
- params.filter.host с минимум одним host name.

Tags не создаются отдельным справочником Zabbix. Они передаются напрямую в host.create/host.update в поле params.tags.
Для tags проверяется наличие непустого поля tag.
Для dynamic host groups `zabbixrequests2api` перед host.create/host.update обязан выполнить `hostgroup.get` по имени. Если group найдена, writer подставляет `groupid`; если group не найдена и `Zabbix:AllowDynamicHostGroupCreate=true`, writer вызывает `hostgroup.create` и подставляет новый `groupid` в тот же host payload, чтобы текущий host сразу попал в созданную group; если создание выключено, writer не вызывает host.create/update и публикует ошибку `auto_expand_disabled`.

14.5. Проверка справочников Zabbix

Перед host.create и host.update сервис должен проверять:
- наличие всех params.groups[].groupid через hostgroup.get;
- наличие всех params.templates[].templateid через template.get;
- наличие template group у найденных templates через template.get с selectTemplateGroups.
- отсутствие конфликтов совместимости в итоговом наборе templates через template.get с selectItems/selectDiscoveryRules: одинаковый item key, LLD rule key или inventory_link в двух и более шаблонах считается блокирующей ошибкой.

Если справочник отсутствует или найден конфликт совместимости templates, сервис не вызывает host.create/host.update, а публикует error response с `zabbixRequestSent=false`. Для `template_conflict` errorMessage должен раскрывать конфликтующий key или inventory link, имена/templateid шаблонов, ожидаемое исправление и место чтения: `PROJECT_DOCUMENTATION.md` / `PROJECT_DOCUMENTATION.en.md`, section `Zabbix template compatibility`.

Коды ошибок:
- missing_host_group;
- missing_template;
- missing_template_group;
- template_conflict;
- invalid_zabbix_request;
- invalid_json;
- zabbix_api_error;
- zabbix_http_error.

14.6. Gentle processing

Сервис не должен отправлять объекты в Zabbix непрерывным потоком.

Между обработкой объектов должна быть настраиваемая задержка:

Processing.DelayBetweenObjectsMs

Значение по умолчанию:

2000 ms

Retry-настройки также задаются через конфиг:
- Processing.MaxRetryAttempts;
- Processing.RetryDelayMs.

{
  "source": "zabbixrequests2api",
  "eventType": "zabbix.host.binding.created|zabbix.host.binding.updated|zabbix.host.binding.deleted",
  "operation": "host.create|host.update|host.delete",
  "sourceClass": "<CMDBuild class>",
  "sourceCardId": "<CMDBuild card id>",
  "sourceCode": "<CMDBuild code или null>",
  "hostProfile": "main|<profile name>",
  "isMainProfile": true|false,
  "zabbixHostId": "<hostid>",
  "zabbixHostName": "<technical host name>",
  "bindingStatus": "active|deleted",
  "rulesVersion": "<rules version>",
  "schemaVersion": "<schema version>",
  "requestId": "<JSON-RPC id>",
  "occurredAt": "<UTC timestamp>",
  "input": {
    "topic": "<input topic>",
    "partition": 0,
    "offset": 0,
    "key": "<input key>"

Kafka key: `sourceClass:sourceCardId:hostProfile`.

Если нет `sourceClass`, `sourceCardId` или `zabbixHostId`, binding event не публикуется, а сервис пишет warning.

14.10. Шаблоны Zabbix

Шаблоны мониторинга не являются JSON-файлами проекта.
Шаблоны мониторинга являются объектами внутри Zabbix.

JSON используется:
- как формат rules-файла конвертации;
- как формат JSON-RPC запроса к Zabbix API.

В JSON-RPC host.create сервис передает только ссылки на уже существующие шаблоны Zabbix:

{
  "templates": [
    { "templateid": "10564" },
    { "templateid": "10001" }
  ]
}

На текущем dev-стенде подтверждены шаблоны:

zabbix.host.bindings

Development log topic:

zabbixbindings2cmdbuild.logs.dev

Production/base log topic:

zabbixbindings2cmdbuild.logs

Health endpoint:

GET http://localhost:5083/health

Назначение:
- читать binding events, опубликованные zabbixrequests2api после успешного host.create/update/delete;
- для `isMainProfile=true` обновлять исходную карточку CMDBuild и писать `zabbix_main_hostid=<hostid>`, а при `host.delete` очищать это поле;
- для дополнительных профилей искать карточку класса `ZabbixHostBinding` по `OwnerClass + OwnerCardId + HostProfile` и создавать или обновлять связь;
- перед `PUT` сравнивать текущее состояние CMDBuild с binding event и пропускать запись, если `zabbix_main_hostid` или существующая карточка `ZabbixHostBinding` уже совпадает;
- начиная со второго этапа обеспечивать данные для прямой идентификации update/delete: `cmdbkafka2zabbix` читает `zabbix_main_hostid` и `ZabbixHostBinding` перед fallback `host.get`;
- invalid JSON пропускать как poison message с фиксацией state/offset;
- ошибки записи в CMDBuild считать retryable и не коммитить Kafka offset.

Конфиг должен содержать секции:
- `Kafka:Input`;
- `Cmdbuild:BaseUrl`, `Cmdbuild:Username`, `Cmdbuild:Password`;
- `Cmdbuild:MainHostIdAttributeName`, default `zabbix_main_hostid`;
- `Cmdbuild:BindingClassName`, default `ZabbixHostBinding`;
- `Cmdbuild:BindingLookupLimit`;
- `ProcessingState`;
- `ElkLogging`.

CMDBuild service account для этого сервиса должен иметь read/update права на карточки классов, участвующих в conversion rules, и read/create/update права на служебный класс `ZabbixHostBinding`.

18. Frontend/BFF monitoring-ui-api

18.1. Назначение

monitoring-ui-api является Node.js компонентом для frontend и backend-for-frontend.

Frontend не должен обращаться напрямую к Kafka, CMDBuild, Zabbix или IdP-интеграционным endpoint.
Все обращения выполняются через monitoring-ui-api.

Реализованный срез:
- статический frontend dashboard;
- Node.js HTTP API;
- server-side session;
- local login form для пользователей UI;
- внешний вход через MS AD LDAP/LDAPS или IdP SAML2/OAuth2/OIDC;
- SAML2 SP endpoints login, ACS, metadata и logout;
- OAuth2 Authorization Code endpoints login/callback;
- read-only health dashboard по health endpoints микросервисов;
- Events view с read-only просмотром настроенных Kafka topics через BFF;
- rules view с загрузкой JSON, validate, dry-run и browser save-as;
- Zabbix catalog sync;
- раздел `Метаданные Zabbix` с templates, item keys, LLD rule keys, inventory bindings, existing host templates и индексом конфликтов templates;
- CMDBuild catalog sync;
- три admin-подменю: `Авторизация` для локального входа/MS AD/IdP, `Runtime-настройки` для подключений, Zabbix API key, AuditStorage и Kafka Events, `Настройка git` для правил чтения conversion rules file.

18.2. Авторизация

Поддерживаются три роли UI:
- Просмотр (`viewer`) - меню `Панель` и `События`;
- Редактирование правил (`editor`) - все меню кроме `Авторизация`, `Runtime-настройки` и `Настройка git`;
- Администрирование (`admin`) - все меню и администрирование пользователей/settings.

Локальные пользователи:
- файл пользователей хранится рядом с runtime settings: `src/monitoring-ui-api/state/users.json`;
- при первом старте создаются `viewer/viewer`, `editor/editor`, `admin/admin`;
- пароли хранятся как PBKDF2-SHA256 hash/salt, plaintext-пароли в файле не сохраняются;
- пользователь может сменить свой пароль после входа;
- администратор может сбросить пароль пользователю через подменю `Авторизация` и потребовать смену при следующем входе.
- при выборе режима `MS AD` или `IdP` блок локальных пользователей в UI должен быть недоступен; роли внешних пользователей назначаются только через соответствие группам.
- при deployment начальные UI-пароли меняются после первого входа или через заранее подготовленный `state/users.json`; пароли CMDBuild/Zabbix по умолчанию не задаются.

Поддерживаются три режима авторизации UI:

Локальная:
- пользователь входит в UI локальной учетной записью из users-файла;
- CMDBuild/Zabbix login/password запрашиваются на server-side session только при первой операции с соответствующим API.

MS AD:
- `Auth:UseIdp=true`, `Idp:Provider=LDAP`;
- login/password вводятся в UI login form и проверяются через LDAP/LDAPS bind;
- найденные AD-группы сопоставляются с ролями `admin`, `editor`, `viewer` по `Idp:RoleMapping`.

IdP:
- `Auth:UseIdp=true`, `Idp:Provider=SAML2` или `OAuth2`;
- IdP выполняет идентификацию пользователя;
- если LDAP/LDAPS service bind настроен, BFF ищет login в MS AD и назначает роли по AD-группам; если AD lookup не настроен, используется прежний fallback по группам из IdP claims;
- внешняя авторизация UI через MS AD или IdP используется только для входа в UI и назначения ролей;
- использовать данные авторизации MS AD/IdP как backend credentials для CMDBuild и Zabbix запрещено: технически целевые системы в нашем контуре не принимают эти данные как API credentials;
- runtime-флаги `Cmdbuild:UseIdp` и `Zabbix:UseIdp` не поддерживаются и не должны отображаться в UI;
- для Zabbix сохраняется `Zabbix:ApiToken`: если он заполнен, backend использует его первым; если он не заполнен, UI при первой операции запрашивает Zabbix login/password и хранит их только в server-side session;
- для CMDBuild backend при первой операции запрашивает CMDBuild login/password и хранит их только в server-side session;
- настройки авторизации вносятся через подменю `Авторизация` после входа admin;
- если группа не совпала с `Idp:RoleMapping`, назначается `viewer`;

SAML2 provider:
- SP metadata публикуется по /auth/saml2/metadata;
- login стартует через /auth/saml2/login;
- ACS принимает HTTP-POST binding по /auth/saml2/acs;
- SAML response валидируется библиотекой @node-saml/node-saml;
- InResponseTo проверяется обязательно;
- IdP signing certificate обязателен через Idp:IdpX509Certificate, Idp:IdpX509CertificatePath или MetadataUrl.

OAuth2/OIDC provider:
- login стартует через /auth/oauth2/login;
- callback принимает code по /auth/oauth2/callback;
- BFF выполняет token exchange и получает claims через UserInfo endpoint или id_token;
- OAuth2 access token, SAML2 identity и MS AD login не передаются в CMDBuild/Zabbix как backend credentials.

LDAP/LDAPS/MS AD provider:
- login/password вводятся в UI login form и проверяются через LDAP bind;
- поддерживаются service bind DN/password или bind по `UserDnTemplate`;
- user/group filters используют плейсхолдеры `{login}` и `{dn}`;
- группы читаются из `memberOf`/настроенного groups attribute и/или через group search filter;
- найденные группы сопоставляются с ролями `admin`, `editor`, `viewer` по `Idp:RoleMapping`.

IdP disabled / локальный режим:
- пользователь входит в UI локальной учетной записью из users-файла;
- CMDBuild login/password запрашиваются при первой операции, которой нужен CMDBuild API, и хранятся только в памяти server-side session;
- Zabbix использует runtime `Zabbix:ApiToken`, если он заполнен;
- если Zabbix API key не заполнен, Zabbix login/password запрашиваются при первой операции, которой нужен Zabbix API, и хранятся только в памяти server-side session;
- production secrets не попадают в git.
- сервисные учетные записи микросервисов могут получать пароль/token через Indeed PAM/AAPM: в конфигурации хранится `secret://id`, а фактическое значение считывается при старте сервиса только в память процесса; AAPM application token или application login/password является bootstrap-секретом deployment-слоя и может передаваться через `PAMURL`/`PAMUSERNAME`/`PAMPASSWORD`.

CMDBuild/Zabbix credentials при MS AD/IdP:
- внешний вход через MS AD/IdP используется только для UI identity и ролей;
- Zabbix API key имеет приоритет и используется первым, если он задан;
- CMDBuild login/password и, при отсутствии Zabbix API key, Zabbix login/password запрашиваются на текущую session;
- постоянные CMDBuild/Zabbix login/password в config/runtime-файлах не хранятся.

Минимальные права по операциям:
- CMDBuild user для UI/catalog sync должен иметь REST API login и read-only доступ к metadata classes/attributes/domains, lookup types/values, relations текущих карточек и карточкам целевых классов, включая reference/domain-цепочки, которые используются в `source.fields[].cmdbPath`; права create/update/delete на карточки CMDBuild для catalog sync не требуются;
- CMDBuild user для `Настройка webhooks` должен иметь read-доступ к ETL/webhook records через REST v3 `/etl/webhook/?detailed=true`, чтобы выполнять `Загрузить из CMDB` и сравнивать текущее состояние с rules;
- CMDBuild user для `Загрузить в CMDB` должен иметь create/update/delete или эквивалентные modify-права на ETL/webhook records через REST v3 `/etl/webhook/`; эти права нужны только оператору, который применяет webhook-план в CMDBuild;
- CMDBuild service account для `cmdbkafka2zabbix` должен иметь read-only доступ к исходным карточкам и `ZabbixHostBinding`, если включено `Cmdbuild:HostBindingLookupEnabled`;
- CMDBuild service account для `zabbixbindings2cmdbuild` должен иметь read/update права на карточки классов, участвующих в conversion rules, чтобы записывать/очищать `zabbix_main_hostid`, а также read/create/update права на класс `ZabbixHostBinding`;
- backend обязан ограничивать запись managed-префиксом `cmdbwebhooks2kafka-*`, но это не заменяет CMDBuild permission model: учетная запись CMDBuild должна быть ограничена настолько узко, насколько позволяет CMDBuild;
- Zabbix user/API token для UI/catalog sync должен иметь API access и read-only доступ к используемым host groups, template groups, templates, hosts/tags и справочникам, читаемым через `*.get` методы;
- права host create/update/delete для чтения каталогов не требуются, но нужны отдельному Zabbix API пользователю сервиса `zabbixrequests2api`, который применяет мониторинг.

18.3. Rules local file save

Frontend должен уметь загружать и сохранять JSON rules-файл через браузер без записи active rules-файла на backend.

Поток:
- пользователь выбирает локальный JSON;
- frontend отправляет JSON в monitoring-ui-api;
- API валидирует JSON;
- API может выполнить dry-run на тестовом CMDB payload;
- при сохранении frontend скачивает JSON-файл через браузер;
- пользователь самостоятельно кладет файл в удобный git repository и публикует изменения вне приложения;
- при включенной `Настройка git` frontend может записать rules JSON в локальную working copy вместе с соседним `*.webhooks.json`, но commit/push все равно остаются ручной операцией оператора;
- после публикации оператор нажимает `Перечитать правила конвертации`, а `cmdbkafka2zabbix` подтягивает/перечитывает rules по своей `ConversionRules` настройке.

Local file save, dry-run, validate, mapping и catalog sync доступны ролям `editor` и `admin`.
`Авторизация`, `Runtime-настройки`, `Настройка git` и сброс пароля доступны только роли `admin`.

Подменю `Настройка git` должно быть отделено от `Runtime-настройки`. Оно хранит в UI settings только блок `rules`: `RulesFilePath`, `ReadFromGit`, `RepositoryPath`, `RepositoryUrl`, показывает resolved path, `schemaVersion`, `rulesVersion` и умеет загрузить текущий файл правил из выбранного источника. UI не выполняет commit/push; публикация файла остается задачей оператора. При включенном `ReadFromGit` внутри repository ожидается файл с именем/путем из `RulesFilePath`, по умолчанию `rules/cmdbuild-to-zabbix-host-create.json`. Кнопка `Сохранить в git` записывает rules JSON с новым `rulesVersion` в локальную working copy и рядом формирует согласованный webhook artifact `*.webhooks.json`; token/password/secret/API key/Authorization значения в webhook artifact должны быть заменены на `XXXXX`.

Основные секции:

Runtime settings:

- `AuditStorage` должен поддерживать provider `postgresql` и `sqlite`, connection string, PostgreSQL schema, `AutoMigrate`, `CommandTimeoutSeconds`; dev config использует SQLite. SQLite разрешен для небольших инсталляций с ориентиром до 1000 объектов на мониторинге, допустимо до 2000 объектов при умеренном потоке событий и коротком хранении аудита; для большего объема, высокой параллельности пользователей или длительного хранения аудита требуется PostgreSQL. Код раздела аудита должен работать через общий storage abstraction и проверяться на обеих СУБД;
- подменю `Авторизация` должно позволять выбирать один из режимов `Локальная`, `MS AD`, `IdP`, редактировать SAML2/OAuth2 и LDAP/LDAPS/MS AD параметры, маппинг ролей к группам и локальных пользователей;
- сохранение выполняется в файл `src/monitoring-ui-api/state/ui-settings.json`;
- runtime state/settings не попадают в git;
- текущий dev-контур должен быть заполнен значениями `CMDBuild localhost:8090`, `Zabbix localhost:8081`, `Kafka localhost:9092`, topics `*.dev`.

Audit model:
- UI должен иметь раздел `Аудит`, доступный ролям `editor` и `admin`; проверка модели доступна `editor/admin`, применение изменений в CMDBuild доступно только `admin`;
- проверка audit model должна по conversion rules определить участвующие классы CMDBuild и построить план без модификации CMDBuild;
- применение audit model должно добавить в участвующие классы отсутствующий атрибут `zabbix_main_hostid`. Этот атрибут хранит Zabbix `hostid` основного host для конкретной карточки CMDBuild и нужен для аудита/диагностики постановки карточки на мониторинг;
- `zabbix_main_hostid` относится только к основному host profile. Для дополнительных `hostProfiles`, когда одна карточка CMDBuild создает несколько Zabbix hosts, используется отдельный класс связей;
- UI должен проверять наличие класса `ZabbixHostBinding`; если его нет, администратор должен выбрать в дереве CMDBuild родительский класс, под которым он будет создан;
- `ZabbixHostBinding` должен хранить одну карточку на связь `CMDBuild class + card id + hostProfile -> Zabbix host`. Минимальные атрибуты: `OwnerClass`, `OwnerCardId`, `OwnerCode`, `HostProfile`, `ZabbixHostId`, `ZabbixHostName`, `BindingStatus`, `RulesVersion`, `LastSyncAt`;
- назначение атрибутов: `OwnerClass`/`OwnerCardId`/`OwnerCode` описывают исходную карточку CMDBuild; `HostProfile` указывает профиль из rules; `ZabbixHostId`/`ZabbixHostName` указывают объект Zabbix; `BindingStatus`, `RulesVersion`, `LastSyncAt` нужны для состояния связи, версии правил и времени последней синхронизации;
- после успешной записи в Zabbix микросервис `zabbixbindings2cmdbuild` должен автоматически заполнять `zabbix_main_hostid` для основного профиля и карточки `ZabbixHostBinding` для дополнительных профилей;
- UI должен предоставлять быстрый аудит как read-only операцию для `editor/admin`: выбор класса CMDBuild с учетом дочерних классов и фильтра только участвующих в rules, чтение карточек через `limit/offset`, расчет ожидаемых host/profile/interface/groups/templates/maintenance/status по текущим conversion rules и сравнение с Zabbix `host.get` и bulk `maintenance.get`. `Запустить быстрый аудит` должен читать текущий offset, `Следующий пакет` должен увеличивать offset на текущий лимит карточек на класс. Быстрый аудит не должен выполнять автоисправления;
- для применения audit model CMDBuild user должен иметь права администратора модели на создание классов и атрибутов через REST (`POST /classes?scope=service`, `POST /classes/{class}/attributes`). Для проверки плана эти write-права не требуются.

Events Kafka browser:
- браузер не подключается к Kafka напрямую;
- `monitoring-ui-api` читает Kafka через backend adapter;
- список topics берется из `EventBrowser:Topics`;
- текущие topics: `cmdbuild.webhooks.dev`, `zabbix.host.requests.dev`, `zabbix.host.responses.dev`, `zabbix.host.bindings.dev`, `cmdbwebhooks2kafka.logs.dev`, `cmdbkafka2zabbix.logs.dev`, `zabbixrequests2api.logs.dev`, `zabbixbindings2cmdbuild.logs.dev`;
- поддерживаются настройки `BootstrapServers`, `ClientId`, `SecurityProtocol`, `SaslMechanism`, `Username`, `Password`, `MaxMessages`, `ReadTimeoutMs`;
- просмотр read-only, topics не создаются;
- по умолчанию выводятся последние сообщения, сортировка от более новых к более старым;
- раскрытый topic показывает timestamp, сводку service/partition/offset/key и value.

20. Уточнения по текущей реализации от 2026-04-30

20.1. Сетевой запуск webhook-сервиса

CMDBuild работает в Docker-контейнере и не может обращаться к `localhost` dev host.

Текущий dev endpoint webhook:

http://192.168.202.100:5080/webhooks/cmdbuild

`cmdbwebhooks2kafka` должен слушать `0.0.0.0:5080`.
Запуск только на `localhost:5080` считается ошибочной конфигурацией для текущего dev-контура, потому что webhook из CMDBuild не дойдет до сервиса.

Healthcheck:

http://localhost:5080/health
http://192.168.202.100:5080/health

20.2. Kafka state и восстановление после рестарта

Для `cmdbkafka2zabbix` и `zabbixrequests2api` state-файл должен использоваться не только для диагностики.

Обязательное поведение:
- при успешной обработке записывать последний input topic, partition, offset и идентификатор объекта;
- при старте читать state-файл;
- после назначения Kafka partition начинать чтение с `lastInputOffset + 1` для сохраненного topic/partition;
- Kafka offset коммитить только после успешной публикации результата или осознанного skip/error response.

Это предотвращает повторную обработку старых сообщений после рестарта при рассинхронизации committed offset consumer group и локального state-файла.

20.3. Zabbix inventory

Если rules/T4 формируют объект `inventory`, то `inventory_mode` не должен быть `-1`.

Причина:
- `inventory_mode=-1` отключает inventory в Zabbix;
- Zabbix отклоняет запрос с ошибкой `Cannot set inventory fields for disabled inventory`, если одновременно переданы `inventory_mode=-1` и `inventory`.

Текущее dev-значение:

inventory_mode = 0

20.4. Расширенная модель rules

Rules JSON должен поддерживать следующие расширения Zabbix host payload:
- proxy;
- proxy group;
- Zabbix interfaces[] profile;
- host status;
- TLS/PSK;
- host macros;
- inventory fields;
- maintenances;
- value maps.

Live E2E для текущего host-create/update контура должен проверять на созданных в Zabbix hosts как минимум:
- technical host name host;
- visible name name;
- interfaces[];
- groups[];
- templates[];
- tags[];
- host macros;
- inventory fields;
- host status;
- TLS/PSK mode.

Zabbix host.get не возвращает PSK secret и может не возвращать PSK identity, поэтому live E2E проверяет примененный TLS mode через tls_connect/tls_accept. Proxy/proxy group требуют заранее созданных proxy objects в Zabbix. Maintenance и value maps относятся к отдельным Zabbix API operation/catalog сценариям, если они не включены в конкретный T4 host payload. Такие сценарии тестируются отдельным setup, а не базовым host-create demo.

T4-шаблоны используют промежуточную модель `Model.*`.
`Model.*` не является прямой схемой CMDBuild или Zabbix. Это объект конвертера `cmdbkafka2zabbix`, который собирается из CMDB payload, rules и catalog references.

Для динамических полей используется:

Model.Field("fieldName")

20.5. UI: управление и логический контроль правил конвертации

В `monitoring-ui-api` должны быть доступны:
- `Управление правилами конвертации`: три колонки CMDBuild -> rules -> Zabbix с подсветкой участвующих элементов;
- lookup mapping по конкретной связке class + lookup + value;
- повторное нажатие на элемент снимает выделение;
- Zabbix catalog sections закрыты по умолчанию и загружаются по `+` или при попадании в выделенную цепочку;
- edit mode управления правилами с действиями `Добавление правила`, `Модификация правила` и `Удаление правила`; при входе в edit mode нижний трехколоночный preview должен скрываться, чтобы оператор работал только с формой изменения draft JSON;
- список CMDBuild classes в edit mode должен показывать hierarchy с визуальными отступами; superclass/prototype classes нельзя выбирать для новых или изменяемых rules, а при попадании уже существующего rule на superclass UI должен подставлять ближайший конкретный subclass;
- добавление rule выбирает CMDBuild class, class attribute field, conversion structure, Zabbix object/payload, priority и regex;
- модификация rule должна начинаться без автоматически выбранного rule; оператор может начать с rule, CMDBuild class, class attribute field или conversion structure, UI фильтрует связанные списки, а если по выбранным фильтрам остается единственное matching rule, оно выбирается автоматически и загружается в ту же форму;
- изменение любого узла формы должно инвалидировать только зависимые нижележащие части: смена class очищает leaf field и Zabbix target, смена field фильтрует conversion structures, смена conversion structure фильтрует fields и Zabbix targets;
- UI должен блокировать сохранение, пока цепочка не доведена до однозначного leaf/source field и совместимого Zabbix target; валидные поля подсвечиваются зеленым, обязательные/конфликтные - красным, загруженные из rule, но не подтвержденные текущим catalog/filter, - желтым;
- в Runtime-настройках должны быть две независимые галки: `Разрешить динамическое расширение Zabbix Tags из CMDBuild leaf` и `Разрешить динамическое создание Zabbix Host groups из CMDBuild leaf`; при выключенной галке `Правило tag`/`Правило host group` требует существующий Zabbix target, при включенной галке UI показывает отдельный target `Создавать/расширять из CMDBuild leaf` и сохраняет `targetMode=dynamicFromLeaf`; для dynamic targets и scalar payload objects редактор add/modify должен показывать блок `Строка payload`: если он пустой, используется выбранный class attribute field, если заполнен - сохраняется `nameTemplate`/`valueTemplate`, который может складывать строку через `Model.Source("field")` и вырезать/заменять части через `Model.Regex("field", "pattern")`/`Model.Regex("field", "pattern", "replacement")` из любых доступных direct/reference/domain/lookup leaf-полей текущего class; для host groups первый приход нового leaf-значения должен создавать group только при разрешенном `Zabbix:AllowDynamicHostGroupCreate`, после чего тот же host.create/update payload должен получить новый `groupid`; Help и Runtime-настройки должны предупреждать, что динамическое расширение включается ответственно, после анализа разнообразия leaf-значений, так как неконтролируемые изменения mapped-атрибутов в CMDBuild создадут такой же объем динамических изменений в Zabbix;
- Zabbix target, загруженный из rule, но отсутствующий в текущем Zabbix catalog/options, должен считаться неконсистентным, подсвечиваться красным и блокировать сохранение; это та же категория ошибки связности, что отсутствующий CMDBuild class/attribute на другой стороне цепочки;
- кнопка `Сбросить поля` должна очищать выбранное rule и все фильтры в режиме модификации, возвращая форму к пустому старту, и очищать leaf field/target в режиме добавления;
- если class attribute field имеет тип `reference`, UI должен итеративно раскрывать target class до читаемых scalar/lookup leaf-полей и сохранять выбранный путь в `source.fields[].cmdbPath`;
- если CMDBuild catalog содержит domain между текущим class и target class, UI должен предлагать path вида `Класс.{domain:СвязанныйКласс}.Атрибут`, поддерживать дальнейшее reference/lookup-погружение и не разрешать связывать потенциально множественное domain-field со скалярной Zabbix structure, кроме fields с явным `resolve.collectionMode=first`;
- удаление rule должно быть tree-based или аналогичным: `Дерево CMDBuild` группирует rules по CMDBuild class и class attribute field, `Дерево Zabbix` группирует rules по Zabbix payload fields, Zabbix object groups и конкретным Zabbix objects, `Дерево rules` группирует rules по коллекциям JSON;
- checkbox на группе удаления должен отмечать все вложенные rules, чтобы оператор мог удалить все rules для конкретного CMDBuild attribute, все rules для конкретного Zabbix attribute/payload field, все rules для Zabbix object group или все rules для класса;
- удаление меняет только выбранные rules из draft JSON и не чистит classes/source fields автоматически;
- undo/redo хранит историю текущей UI-сессии;
- `Save file as` сохраняет draft rules JSON с новым `rulesVersion` и второй текстовый файл для CMDBuild webhook Body/DELETE-инструкций только по добавленным и удаленным в текущей UI-сессии rules/classes/source fields;
- второй webhook-файл должен явно помечать удаления и не смешивать неизмененные правила с изменениями текущей сессии;
- webhook-файл должен показывать path metadata рядом с Body snippets, но реальный CMDBuild payload остается плоским и отправляет только source key со значением/id;
- перед сохранением выполняется проверка: каждый мониторинговый класс из `source.entityClasses` или `className` regex rules должен иметь IP или DNS class attribute field, связанный с `interfaceAddressRules` или `hostProfiles[].interfaces`, и применимый `hostProfiles[]`;
- UI должен иметь отдельный блок `Профили мониторинга` для создания, модификации и удаления `hostProfiles[]`; блок выбирает CMDBuild class, тип `Основной`/`Дополнительный`, IP/DNS leaf, режим IP/DNS, `interfaceProfileRef`, имя profile и `createOnUpdateWhenMissing`;
- добавление/модификация обычного conversion rule не должны скрыто создавать `hostProfiles[]`; если profile для class отсутствует, это фиксируется logical control как риск `no_host_profile_matched`;
- дополнительный profile должен создаваться как отдельный fan-out profile с `hostNameTemplate` с suffix `HostProfileName`, interface `valueField` на выбранный leaf и условием выбора по заполненному leaf или `delete`; имя profile должно проверяться на дубликаты;
- при переименовании profile UI должен обновлять точные regex-условия по виртуальному полю `hostProfile` в связанных rules; при удалении profile UI должен удалять profile и rules, явно ограниченные этим `hostProfile`, но не должен удалять Zabbix host в управляемой системе;
- назначение template/group/tag на созданный дополнительный profile должно выполняться отдельными правилами через виртуальное поле `hostProfile` или через явный scope-чекбокс `Ограничить правило выбранным hostProfile` в форме add/modify; это позволяет оставить основное условие rule на обычном class attribute field, например `description`, lookup или domain leaf, и одновременно добавить allRegex по выбранному `hostProfile`;
- счетчик `Назначения` у hostProfile должен считать только rules, которые явно содержат условие по `hostProfile`; class-level rules без такого условия не считаются назначением на конкретный profile;
- редактор добавления/модификации rules должен показывать виртуальные поля `hostProfile` и `outputProfile`; эти поля заполняются converter для каждого `hostProfiles[]` и не должны записываться в `source.fields` как CMDBuild attributes;
- при добавлении/модификации `interfaceAddress` UI обязан проверять соответствие mode и типа field: IP-looking CMDBuild attribute нельзя сохранять как DNS target `interfaces[].dns/useip=0`, DNS/FQDN-looking attribute нельзя сохранять как IP target `interfaces[].ip/useip=1`, а field с неподтвержденным адресным типом нельзя сохранять как interface address без явного имени/source metadata или `validationRegex`;
- при добавлении/модификации `Правило template` UI обязан использовать `Метаданные Zabbix` и блокировать сохранение, если выбранный template вместе с default/templates после применения `templateConflictRules` оставляет duplicate item key, duplicate LLD rule key или duplicate inventory link;
- имена классов нормализуются: `NetworkDevice` и `Network device` считаются одним классом, UI предпочитает отображаемое имя из CMDBuild catalog;
- `Логический контроль правил конвертации`: без интерактивной подсветки цепочек, только выявление отсутствующих элементов источников и правил;
- checkbox над отсутствующими элементами;
- если класс присутствует в `source.entityClasses`, но не имеет применимого `hostProfiles[]`, это считается ошибкой rules, потому что converter пропустит событие с `no_host_profile_matched`; UI должен предлагать действие `Создать host profile`;
- если template rule оставляет несовместимый итоговый template set после `templateConflictRules`, UI должен показать критическую ошибку с конфликтующим key/link и предложить оператору отредактировать rule, добавить/исправить `templateConflictRules` или удалить ошибочный rule из draft;
- если в текущей UI-сессии загружены CMDBuild webhooks, logical control должен использовать rule-based webhook requirements и показывать предупреждения по отсутствующим managed webhooks или payload-полям, которые нужны rules и не передаются webhook;
- выбор всех/снятие выбора на заголовках;
- удаление выбранных отсутствующих элементов только после подтверждения пользователя;
- после применения выбранных исправлений формировать исправленный JSON в памяти и сохранять его через браузер без записи backend rules-файла; undo/redo работают в общем потоке Logical Control.

20.5.1. UI: настройка CMDBuild webhooks

В `monitoring-ui-api` должен быть раздел `Настройка webhooks`.

Доступ:
- роль `viewer` не видит раздел;
- роли `editor` и `admin` видят раздел и могут работать с планом webhooks;
- CMDBuild login/password для операций спрашиваются на server-side session при первой необходимости, как и для catalog sync.

Требования к поведению:
- пользоваться этим разделом не обязательно: webhooks можно настроить вручную в CMDBuild или использовать webhook-файлы, которые сохраняются вместе с файлом правил конвертации;
- `Загрузить из CMDB` читает текущие CMDBuild ETL/webhook records через REST v3 `/etl/webhook/?detailed=true`;
- после одной только загрузки из CMDB UI должен показывать текущие webhooks справочно, даже если план операций еще не построен;
- `Проанализировать rules` каждый раз заново читает актуальный rules JSON и CMDBuild catalog cache, считает conversion rules источником правды для состава webhook payload, строит `webhook requirements` по используемым source fields и уже из них формирует желаемый набор webhooks, не создавая и не меняя records в CMDBuild;
- если rules используют leaf через lookup/reference/domain, desired webhook должен передавать только ближайший scalar/id текущей карточки: CMDBuild attribute для lookup/reference path или `Id` текущей карточки для domain path; конкретный leaf поднимает converter по metadata `rules.source.fields[].cmdbPath`;
- для отсутствующего managed webhook формируется операция `Создать`;
- для существующего managed webhook с отличиями в body/event/target/method/url/headers/active/language формируется операция `Изменить`;
- для существующего managed webhook с префиксом `cmdbwebhooks2kafka-`, который больше не нужен текущим rules, формируется операция `Удалить`;
- если в существующем CMDBuild webhook отсутствуют payload-поля, которые нужны rules выбранного класса, UI должен явно показывать эти ключи и правила-источники требования в summary, деталях строки и причине операции; это напоминание означает, что без `Загрузить в CMDB` или ручной правки webhook converter не получит соответствующие значения;
- каждая строка таблицы должна позволять раскрыть payload; добавляемые значения показываются зеленым, удаляемые красным, актуальные черным;
- нажатие на значение в столбце `Действие` должно открывать детали под выбранной строкой, а общий блок `Детали` должен находиться под таблицей и подсвечивать текст теми же цветами;
- при анализе существующего managed webhook уже загруженное body должно сохраняться, а optional source fields должны добавляться только если они нужны правилам выбранного класса; добавление независимого класса не должно массово переводить остальные records в `Изменить`;
- анализ не должен добавлять duplicate key с другим регистром или alias, например `OS` рядом с уже существующим `os`;
- `cmdbPath` с корневым классом, не совпадающим с текущим классом webhook, не должен порождать placeholder в body текущего класса;
- каждая строка таблицы должна иметь кнопку `Редактировать`; правка JSON меняет только текущий план и не применяет изменение к CMDBuild до команды `Загрузить в CMDB`;
- для строки текущего CMDB webhook без построенного плана правка должна создавать выбранную `update`-операцию;
- операции удаления не должны быть выбраны по умолчанию, чтобы оператор явно подтвердил удаление старых hooks;
- должна быть отдельная кнопка `Удалить выбранные`, которая применяет только отмеченные операции `Удалить` и не отправляет операции `Создать`/`Изменить`; остальные изменения применяются общей командой `Загрузить в CMDB`;
- undo/redo должны работать с выбором операций текущей UI-сессии;
- undo/redo не должны обещать и выполнять откат уже выполненной загрузки конфигурации в CMDBuild;
- `Сохранить файл как` выгружает JSON-план webhooks через браузер и не меняет CMDBuild, backend rules-файл или git;
- `Загрузить в CMDB` после подтверждения применяет только выбранные операции и действительно меняет CMDBuild через REST v3 `/etl/webhook/`;
- backend должен ограничивать apply workflow управляемым префиксом `cmdbwebhooks2kafka-`, чтобы не удалить или изменить чужие CMDBuild webhooks;
- для `Изменить` и `Удалить` backend перед записью должен перечитывать текущие CMDBuild webhooks через `/etl/webhook/?detailed=true` и выбирать целевой record по managed `code`; `current.id` из browser payload не должен использоваться как доверенный идентификатор целевой записи;
- для новых или изменяемых hooks body остается плоским: reference/domain/lookup leaf path хранится в `rules.source.fields[].cmdbPath`, а CMDBuild webhook передает scalar/id текущего атрибута или id текущей карточки для domain path.

Для проверки полноты редактора правил ведется отдельный план `TEST_PLAN_MAPPING_EDITOR.md`. Тестовая модель строится под абстрактным `CI` / `КЕ` и использует scripts:
- `node scripts/cmdbuild-demo-schema.mjs --apply` - schema classes/lookups/domains;
- `node scripts/cmdbuild-demo-instances.mjs --apply` - повторяемые demo-карточки с описанием проверяемого сценария.
- `node scripts/cmdbuild-demo-e2e.mjs --apply` - полный demo-прогон через webhook -> Kafka -> converter -> Zabbix API с отчетом в `reports/`, включая live-проверку назначений host, name, host groups, templates, interfaces, tags, macros, inventory, status и TLS mode.

Минимальный порядок демонстрации:
1. `node scripts/cmdbuild-demo-schema.mjs --apply`
2. `node scripts/cmdbuild-demo-instances.mjs --apply`
3. `node scripts/cmdbuild-demo-e2e.mjs --apply`

Флаг `--cleanup-zabbix` у E2E удаляет старые demo-hosts `cmdb-c2mtestci-*` перед прогоном и не нужен, если оператор очищает тестовые CMDBuild/Zabbix вручную.

Рабочий demo/e2e rules-файл остается `rules/cmdbuild-to-zabbix-host-create.json`, и активный `RulesFilePath` по умолчанию указывает на него. Базовый набор собран вокруг C2MTest-модели, но в тестовой среде в него можно добавлять любые конкретные классы текущего CMDBuild catalog как smoke-проверки динамической модели. Это не означает жесткой привязки продукта к этим именам: для каждого добавленного класса должны быть `source.entityClasses`, source field для IP/DNS leaf, применимый `hostProfiles[]` и webhooks. Чистый no-op starter для production лежит рядом: `rules/cmdbuild-to-zabbix-host-create.production-empty.json`. Чистый no-op starter для dev лежит рядом: `rules/cmdbuild-to-zabbix-host-create.dev-empty.json`; он создан из пустого installation profile, но заранее содержит `cmdbuild.webhooks.dev` и `http://localhost:8081/api_jsonrpc.php`. Перед использованием starter-файла оператор заполняет source fields, hostProfiles, Zabbix catalog IDs, T4 templates и включает `publish=true` только для проверенных routes. Для работы именно с пустым dev starter оператор загружает его в UI или указывает путь в `Rules.RulesFilePath`.

В UI раздела `Правила` должна быть доступна кнопка `Создать пустой`: она формирует production starter из текущего runtime окружения и catalog cache, но не сохраняет его активным rules-файлом на backend. Перед генерацией backend обязан проверить оба cache: CMDBuild cache должен содержать классы и атрибуты, Zabbix cache должен содержать host groups и templates. Если один из cache отсутствует или пустой, backend возвращает ошибку и starter не формируется. Сгенерированный JSON сохраняется через браузер, после чего оператор самостоятельно публикует файл в rules git.
